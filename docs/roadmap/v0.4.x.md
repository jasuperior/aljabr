# v0.4.x — Reactive component props

## Problem

In v0.3.x, the shorthand reactive prop syntax (`{ class: signal }`) was deliberately scoped to **host elements only**. On a `<div>` or `<span>`, the renderer owns the update cycle: it detects a readable value in a prop, wraps it in a per-prop effect, and calls `setProperty` surgically when the signal changes. The element never re-runs; only the affected attribute updates.

At the **component boundary**, the story is different. Props are passed as a plain record to the component function, which runs exactly once. If you write:

```ts
view(Counter, { count: mySignal })
```

`Counter` receives `mySignal` as a `Signal<number>`, not a `number`. The component's prop type says `count: number`, so TypeScript errors. And even if the type matched, the component body has no mechanism to re-evaluate when the signal changes — components do not re-run.

The v0.3.x answer to this is to wrap the component invocation in a reactive region:

```ts
() => view(Counter, { count: mySignal.get() })
```

This works and the cost is explicit: the entire component subtree tears down and rebuilds on each change. But it requires the call site to know the implementation detail, and it looks inconsistent with the element shorthand.

## What we want

A way to pass a signal as a component prop such that:

1. The component body sees the **resolved type** (`number`, not `Signal<number>` or `() => number`)
2. TypeScript is satisfied — no explicit `Signal<T>` in the component's prop types
3. The component stays as a pure function of props — no internal `.get()` calls needed

## Design options considered

### Option A — Proxy-based props (Solid.js model)

Wrap the props record in a `Proxy` before passing it to the component. Accessing `props.count` transparently calls `signal.get()` and registers a dependency. TypeScript sees `count: number` via the declared prop type; at runtime the getter fires reactively.

**Tradeoff:** Requires `Proxy` support, adds a non-trivial abstraction layer to every component call, and changes the mental model. Props are no longer a plain object — reading them has side effects.

### Option B — Re-running components

Detect Signal props at the `view()` call site and wrap the entire component invocation in a reactive region automatically. The component re-runs when any bound prop changes.

**Tradeoff:** Full subtree teardown on each change. Expensive, and it silently hides that cost. The explicit `() => view(Component, { prop: signal.get() })` pattern already provides this — it should stay explicit.

### Option C — `$prop` binding prefix

A naming convention: `{ $count: mySignal }` means "bind `count` reactively." The renderer strips the `$`, unwraps the value before passing to the component, and re-runs the component when the signal changes.

**Tradeoff:** Still hides a full remount. The `$` prefix is novel syntax with no TypeScript support — the component declares `count: number` but the parent passes `$count: Signal<number>`, and the type system can't connect them.

## Recommendation

**Proxy-based props (Option A)**, implemented carefully:

- Only applied to component calls that receive at least one Signal prop
- Props proxy is transparent to component authors — they declare and use plain types
- Granular tracking: only the prop whose signal changed triggers re-evaluation, not the whole component
- TypeScript: component prop types remain plain; a mapped type utility (`ReactiveProps<P>`) can describe what callers may pass

This requires non-trivial implementation work and a clear ownership model for the proxy lifetime. It is intentionally deferred until the simpler shorthand and list ergonomics from v0.3.x are stable and the component reactivity semantics can be designed without constraint.

## Prerequisite

Before implementing, the advanced guide entry for **reactive props passed into components** (deferred from v0.3.x) should be written. It documents the current explicit pattern and sets user expectations ahead of the v0.4.x change.
