# Aljabr Roadmap (v0.3.5): RefArray Hardening & DerivedArray Rename

_Patch release following v0.3.4. Addresses type-system gaps and missing array methods surfaced by real-world usage of `RefArray` in application code. Also renames `ReactiveArray` to `DerivedArray` — a hard-break rename with no deprecation alias._

---

## Motivation

v0.3.4 shipped `RefArray` as the primary reactive array primitive. First-party application code (a todo app built against the published npm package) exposed two classes of issues:

1. **Type coercion gap** — `Ref.create<Task[]>([])` resolves the overload with `T = Task[]`, producing `RefArray<Task[]>` instead of the intended `RefArray<Task>`. The workaround (`Ref.create([] as Task[])`) is non-obvious and easy to miss.
2. **Incomplete method surface** — the mutation and query API is missing methods that authors reach for naturally: `set`, `find`, `findIndex`, `findLastIndex`, `includes`, `join`, `reduce`, `reduceRight`, `shift`, `unshift`.
3. **Confusing dual-array naming** — `ReactiveArray` and `RefArray` share nearly identical query surfaces but have names that don't communicate their relationship or the distinction between mutable source and derived view. Authors reaching for `map`/`filter`/`sort` on a `RefArray` receive a `ReactiveArray` with no clear signal that it is read-only or computed.

This release closes both gaps. It also migrates the destructive read methods (`pop`, `shift`, `Ref.pop`) from `T | undefined` to `Option<T>` — a deliberate breaking change that aligns them with the library's error-handling philosophy.

---

## Design Decisions

### Type coercion: `Ref.create<T[]>` → `RefArray<T>`

Add a new overload as the first candidate in the overload chain:

```ts
static create<T extends unknown[]>(initial: T): RefArray<T[number]>;
```

When `T = Task[]` is supplied explicitly, `T[number] = Task`, giving `RefArray<Task>`. TypeScript picks this overload first because `T extends unknown[]` is more specific than the existing `T extends object` overloads. The existing overloads and the call-site workaround (`Ref.create([] as Task[])`) remain valid — this is purely additive.

---

### `set(index, value): Option<T>`

Replace-in-place at a specific index. Fine-grained: dirties only the per-index signal for `index`, leaving all other indices and the length signal untouched.

Out-of-bounds indices return `Option.None` rather than throwing or silently no-opping:

- **No throw** — runtime errors leaking into application code violate the library's design contract.
- **No silent no-op** — invisible failures are harder to debug than explicit ones.
- **`Option<T>` return** — `Some(oldValue)` on success, `None` if the index does not exist. The return type communicates fallibility at authoring time; the author is nudged to handle both branches. The old value is returned as a practical bonus for undo/replace patterns.

```ts
match(tasks.set(idx, updatedTask), {
    Some: (prev) => console.log("replaced", prev),
    None: ()     => console.warn("index gone — reactive update may have changed length"),
});
```

`set` does not extend the array. To append, use `push`; to insert at position, use `splice`.

---

### Reactive tracked reads: `find`, `findIndex`, `findLastIndex`, `includes`

All four use **precise dependency tracking**: they call `this.get(i)` for each element they visit, registering per-index signal dependencies, and stop at the first match. Elements beyond the match point are not tracked.

Precise tracking is both more efficient and semantically correct:

- An unvisited element can only affect the result if a visited element first changes (stops matching), which triggers a re-run anyway.
- If no visited element changes, the result is guaranteed unchanged regardless of what unvisited elements do.

Return types use `Option` rather than `T | undefined` or `-1` sentinels:

| Method | Return |
|---|---|
| `find(predicate)` | `Option<T>` |
| `findIndex(predicate)` | `Option<number>` |
| `findLastIndex(predicate)` | `Option<number>` |
| `includes(value)` | `boolean` (precise tracking, stops at first match) |

These are synchronous tracked reads — the same pattern as `get(i)` and `length()`. Call them inside `derived()` or `effect()` to register reactive dependencies.

---

### Reactive tracked reads: `join`, `reduce`, `reduceRight`

Full-array reactive reads: call `this.get(i)` for every index up to `length()`, tracking all per-index signals and the length signal. Any change to any element or the array size triggers re-evaluation.

```ts
derived(() => tasks.join(", "))        // re-runs when any element changes or array grows/shrinks
derived(() => tasks.reduce((sum, t) => sum + t.value, 0))
```

Signatures mirror the native `Array` prototype:

```ts
join(separator?: string): string
reduce<U>(fn: (acc: U, item: T, index: number) => U, initial: U): U
reduceRight<U>(fn: (acc: U, item: T, index: number) => U, initial: U): U
```

---

### Destructive reads: `pop`, `shift`, `Ref.pop` → `Option<T>`

Destructive reads from a possibly-empty container are the clearest case for `Option`: the absence of a value is semantically meaningful, not an error, and callers should be required to handle it.

The following existing and new methods are migrated:

| Method | Old | New |
|---|---|---|
| `RefArray.pop()` | `T \| undefined` | `Option<T>` |
| `RefArray.shift()` *(new)* | — | `Option<T>` |
| `Ref.pop(path)` | `ArrayItem<T,P> \| undefined` | `Option<ArrayItem<T,P>>` |

`unshift` remains `void` — it is purely additive and cannot fail.

---

### `ReactiveArray<T>` → `DerivedArray<T>` (hard-break rename)

`ReactiveArray` is renamed to `DerivedArray`. The name mirrors the existing `Signal`/`Derived` split in the library:

| Mutable | Derived |
|---|---|
| `Signal<T>` | `Derived<T>` |
| `RefArray<T>` | `DerivedArray<T>` |

`DerivedArray` is the type returned by `map`, `filter`, and `sort` on both `RefArray` and `DerivedArray`. It is read-only — it has no mutation methods. Authors cannot construct one directly; it is always the product of a computation over a `RefArray`.

No deprecation alias for `ReactiveArray` is provided. This is a pre-1.0 library and a clean break is preferable to carrying a stale export. Consumers upgrading from v0.3.4 replace `ReactiveArray` with `DerivedArray` at every import and type annotation site — a mechanical find-and-replace.

`IteratorOptions<T>` is unchanged.

---

### `get()` and `peek()` — whole-value reactive reads

All three array/object containers expose two new no-argument overloads alongside their existing indexed/path signatures:

| Call | Type | Behaviour |
|---|---|---|
| `RefArray.get()` | `T[]` | Reactive — tracks a root signal notified on every mutation |
| `RefArray.get(i)` | `T \| undefined` | Reactive — tracks the per-index signal (unchanged) |
| `RefArray.peek()` | `T[]` | Untracked snapshot |
| `RefArray.peek(i)` | `T \| undefined` | Untracked snapshot at index |
| `DerivedArray.get()` | `T[]` | Reactive — tracks a dedicated `#rootSignal` fired on every re-computation |
| `DerivedArray.get(i)` | `T \| undefined` | Reactive — per-index signal (unchanged) |
| `DerivedArray.peek()` | `T[]` | Untracked snapshot |
| `DerivedArray.peek(i)` | `T \| undefined` | Untracked snapshot at index |
| `Ref.get()` | `T \| undefined` | Reactive — tracks the root signal at path `""`, notified on any mutation |
| `Ref.get(path)` | `PathValue<T,P> \| undefined` | Reactive — per-path signal (unchanged) |
| `Ref.peek()` | `T \| undefined` | Untracked snapshot |
| `Ref.peek(path)` | `PathValue<T,P> \| undefined` | Untracked snapshot at path |

**Tracking granularity:** `get()` with no arguments is deliberately coarse — it re-runs whenever anything in the container changes. Authors who need fine-grained subscriptions should supply an index/path, or use iterator methods (`filter`, `find`, `reduce`, etc.).

**`DerivedArray` root signal:** `DerivedArray` had no pre-existing root signal (only per-index and length signals). A private `#rootSignal: Signal<T[]>` is added, set in `#applyUpdate` on every re-computation, and disposed alongside the other signals.

**`RefArray` and `Ref` root signal:** Both already had a signal at the array/object root path (`""` for root `Ref`, `prefix` for `RefArray`) that is notified by `applyArrayMutation` and `#notifyRelated` on every write. No new signal is needed — `get()` with no args simply reads that existing signal.

**`peek()` mirrors `get()`:** The overload signatures of `peek()` are identical to `get()` but wrapped in `untrack()`. This matches the convention established by `Signal.peek()`.

---

### Intentionally retained: `get(i)`, `at(i)`, `Ref.get(path)`

`RefArray.get(i)`, `DerivedArray.get(i)`, and their `at(i)` counterparts remain `T | undefined` and `Derived<T | undefined>` respectively. These are high-frequency reactive read primitives called inside `derived()` and `effect()` on every render cycle, and used internally by `map`, `filter`, `sort`, `find`, `reduce`, and others. Wrapping their return in `Option` would require unwrapping at every internal callsite and add meaningful ceremony to the most common authoring pattern (`tasks.get(0)?.name`).

`Ref.get(path)` is similarly retained: `PathValue<T, P>` is already a deep conditional type, and the `isUnset` property provides a distinct signal for the "no initial value" case.

These are **deliberate holdouts**, not deferred work.

## Scope

| Change | Type |
|---|---|
| `Ref.create<T extends unknown[]>` overload | Type fix |
| `RefArray.set(index, value): Option<T>` | New method |
| `RefArray.find(predicate): Option<T>` | New method |
| `RefArray.findIndex(predicate): Option<number>` | New method |
| `RefArray.findLastIndex(predicate): Option<number>` | New method |
| `RefArray.includes(value): boolean` | New method |
| `RefArray.join(separator?): string` | New method |
| `RefArray.reduce(fn, initial): U` | New method |
| `RefArray.reduceRight(fn, initial): U` | New method |
| `RefArray.shift(): Option<T>` | New method |
| `RefArray.unshift(...items): void` | New method |
| `RefArray.pop(): Option<T>` | **Breaking change** (was `T \| undefined`) |
| `Ref.pop(path): Option<ArrayItem<T,P>>` | **Breaking change** (was `ArrayItem<T,P> \| undefined`) |
| `ReactiveArray<T>` → `DerivedArray<T>` | **Breaking rename** — no deprecation alias |
| `RefArray.get(): T[]` | New overload |
| `RefArray.peek() / peek(i)` | New method |
| `DerivedArray.get(): T[]` | New overload |
| `DerivedArray.peek() / peek(i)` | New method |
| `Ref.get(): T \| undefined` | New overload |
| `Ref.peek() / peek(path)` | New method |

`Signal`, `Derived`, and all schema/ui modules are unchanged.

## Future Work

### `get(i)` / `at(i)` Option ergonomics

The decision to keep `get(i): T | undefined` is load-bearing for authoring ergonomics today. If a future version introduces a more concise unwrap idiom (e.g. a pipeline operator or a dedicated `Option.from(nullable)` bridge), revisiting these as `Option<T>` becomes more attractive. Revisit before v0.4.0.
