# Aljabr Roadmap (v0.3.1): Reactive Iterators & `RefArray`

_Patch release following v0.3.0. Focused on closing the root-array gap in `Ref` and introducing reactive iterator methods that return fine-grained, per-index reactive arrays._

---

## Motivation

`Ref<T>` today assumes the root value is always an object. Array mutation methods (`push`, `pop`, `splice`, `move`) require a path argument typed as `ArrayPath<T>` â€” which generates no valid paths when `T` itself is an array. `Ref.create([1, 2, 3])` compiles, but none of the array methods are callable. That's a gap.

Beyond the gap, there's no way to derive a transformed view of an array that stays reactive. `Derived.create(() => ref.get("items").filter(...))` re-invalidates all downstream subscribers on every source change â€” even if only one element changed. For UI-heavy use cases (lists, tables, grids), this is a meaningful performance problem.

---

## Design Decisions

### `RefArray<T>` â€” new type, read-write

`Ref.create(T[])` returns `RefArray<T>` instead of `Ref<T[]>`. `Ref.at(path)` also returns `RefArray<PathValue<T, P>[number]>` when the resolved path type is an array (currently returns `Ref<array & object>`).

`RefArray<T>` exposes:

```ts
// Pathless root mutations
refArray.push(...items: T[]): void
refArray.pop(): T | undefined
refArray.splice(start: number, deleteCount: number, ...items: T[]): void
refArray.move(from: number, to: number): void

// Per-index reactive reads
refArray.get(i: number): T | undefined        // tracked â€” registers dependency
refArray.at(i: number): Derived<T | undefined> // explicit handle

// Reactive length â€” dedicated signal, notified when array size changes
refArray.length(): number

// Iterators â€” return ReactiveArray<U>
refArray.map<U>(fn: (item: T, i: number) => U): ReactiveArray<U>
refArray.filter(fn: (item: T, i: number) => boolean, opts?: IteratorOptions<T>): ReactiveArray<T>
refArray.sort(comparator: (a: T, b: T) => number, opts?: IteratorOptions<T>): ReactiveArray<T>
```

Pathed nested array mutations (`push(path, ...items)`, etc.) are still supported for nested arrays inside `T` â€” `RefArray` inherits those from `Ref<T>`.

---

### `ReactiveArray<T>` â€” new type, read-only

Returned by all iterator methods. Exposes the same reactive read surface as `RefArray`, plus its own iterator methods for chaining. Mutation is intentionally absent â€” the semantics of writing back through a `filter` or `sort` boundary are undefined.

```ts
reactiveArray.get(i: number): T | undefined
reactiveArray.at(i: number): Derived<T | undefined>
reactiveArray.length(): number

// Chainable
reactiveArray.map<U>(fn: (item: T, i: number) => U): ReactiveArray<U>
reactiveArray.filter(fn: (item: T, i: number) => boolean, opts?: IteratorOptions<T>): ReactiveArray<T>
reactiveArray.sort(comparator: (a: T, b: T) => number, opts?: IteratorOptions<T>): ReactiveArray<T>
```

---

### Key functions & identity tracking

`map` is always 1:1 â€” source index `i` maps to output index `i`. No identity needed.

`filter` and `sort` break index correspondence. Without identity tracking, any source mutation would invalidate all downstream per-index subscribers. To enable surgical, per-index invalidation, both accept an optional key function:

```ts
type IteratorOptions<T> = {
    key?: (item: T) => unknown
}
```

**Default key:** `item => item` (value identity). Works correctly for primitive arrays. For object arrays, this defaults to reference equality â€” which breaks under `Ref`'s immutable update model (every `patch` creates a new object reference). Users should always provide a key for object arrays.

**Enforcement:** Optional at the type level across the board (a type-level conditional enforcement is unreliable for union element types). At runtime in dev mode, a warning is emitted when:
- No key is provided
- Two items produce the same key (duplicates â†’ ambiguous identity)

**Keys are per-step, not inherited.** Each `filter` or `sort` in a chain is responsible for its own identity contract. `map` never needs a key.

**Incremental update algorithm (filter/sort):**
1. On source mutation, re-run the filter/sort function over the new source array
2. Diff the new keyed output against the previous keyed output
3. Notify only the per-index signals whose output item changed (by key) or whose position shifted
4. Update the length signal if output length changed

---

### `length()` implementation

`"length"` is not a valid `Path<T[]>` â€” `PathsOf` only generates `${number}` index paths for arrays. Both `RefArray` and `ReactiveArray` expose `.length()` as a method (consistent with `.state()`, `.get()`) backed by a dedicated `Signal<number>`.

For `RefArray`, `#applyArrayMutation` explicitly notifies the length signal when `newArr.length !== oldArr.length`.

For `ReactiveArray`, the length signal is updated as part of the incremental diff step.

---

### `Ref.at()` update

`at(path)` currently returns `Ref<PathValue<T, P> & object>` for object-typed paths. The return type is extended:

- Path resolves to a **plain object** â†’ `Ref<PathValue<T, P>>`
- Path resolves to an **array** â†’ `RefArray<ArrayItem<T, P>>`
- Path resolves to a **primitive** â†’ `Derived<PathValue<T, P> | undefined>` (unchanged)

---

## API placement

Iterator methods (`map`, `filter`, `sort`) live exclusively on `RefArray` and `ReactiveArray`. They are not added to `Ref<T>` with a path argument â€” the `RefArray` split was motivated precisely by that kind of API incoherence. Users access nested array iterators via `.at(path)`:

```ts
const state = Ref.create({ items: [1, 2, 3, 4, 5] })

const evens = state.at("items")          // â†’ RefArray<number>
    .filter(x => x % 2 === 0)           // â†’ ReactiveArray<number>

const doubled = state.at("items")
    .filter(x => x % 2 === 0)
    .map(x => x * 2)                    // â†’ ReactiveArray<number>
```

## Scope

| Feature | Status |
|---|---|
| `RefArray<T>` â€” pathless root mutation methods | Included |
| `Ref.create(T[])` returns `RefArray<T>` | Included |
| `Ref.at(path)` returns `RefArray` for array paths | Included |
| `ReactiveArray<T>` â€” read-only, per-index reactive | Included |
| `map` / `filter` / `sort` on `RefArray` and `ReactiveArray` | Included |
| Chainable iterators | Included |
| Key-based incremental diffing | Included |
| Dev-mode warning for missing key on object arrays | Included |
| `reduce`, `flatMap`, `find` | Deferred to v0.3.2 or later |
| Two-way writable derived arrays | Out of scope |
