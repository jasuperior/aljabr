### **Aljabr Future Roadmap (v0.3.0): Formal Overview**

**Strategic Vision**

Aljabr currently provides an exceptionally tight, zero-dependency pipeline for defining Algebraic Data Types (ADTs), utilizing non-enumerable symbol tags for clean serialization, and mapping these domain models directly to reactive UI state via the `aljabr/prelude`. While it excels at enforcing compile-time exhaustiveness and eliminating boilerplate for internal data models, as applications scale, developers encounter friction at the boundaries of their domain (e.g., untyped API payloads) and within complex, long-running asynchronous effects.

This roadmap outlines a progressive, six-phase expansion for Aljabr. By drawing inspiration from best-in-class libraries like `ts-pattern` for structural ergonomics, `@effect/schema` for data validation, the `Effect` ecosystem for enterprise-grade resource and defect management, and `SolidJS` for reactive ergonomics, Aljabr can bridge the gap between frontend state machines and robust, fault-tolerant application architectures. The goal is to retain Aljabr’s lightweight, tag-first philosophy while delivering the primitives necessary for deep structural extraction, bidirectional data validation, dependency injection, resilient async lifecycles, and familiar reactive workflows.

### Phase 1: Deep Structural Matching & Ergonomics

_The immediate goal is to make Aljabr's `when()` arms as expressive as best-in-class pattern matchers, without losing the performance benefits of prototype-bound symbol tags._

- **Deep Structural Extraction (`select` bindings):** Currently, pulling nested data out of an Aljabr variant requires manual destructuring in the handler. The roadmap should introduce an extraction utility akin to `ts-pattern`'s `P.select()`, which allows developers to extract a specific nested property directly within the pattern and inject it as a typed argument into the handler.
- **Wildcards and Combinators:** Expand the pattern matching API to support robust combinators such as `P.union`, `P.intersection`, and `P.not`. Adding primitives like `P.string` or `P.number` will allow developers to match variants based on both identity and the shape of their internal payloads simultaneously.
- **Align with TC39 Pattern Matching:** As the ECMAScript Pattern Matching proposal advances, ensure Aljabr's matcher is positioned to smoothly interoperate with or compile down to native `match` expressions and "extractor patterns".

### Phase 1.5: Type-Level Inference for Combinators and `select`

_Deferred from Phase 1. Runtime behavior ships first; type-level precision follows._

- **Combinator Narrowing:** `is.not(is.string)` should narrow the matched value's type to `Exclude<T, string>`. `is.union(is.string, is.number)` should narrow to `string | number`. This requires type-level recursion over combinator arguments to compute the inferred type. Phase 1 ships these with `unknown` as the narrowed type; Phase 1.5 promotes them to precise conditional-type inference.
- **`select` Inference:** Given a pattern containing `select("name")` at any depth, the `selections` second argument to the handler should be typed as `{ name: FieldType }` rather than `Record<string, unknown>`. This requires type-level traversal of the pattern type to collect all `Select<Name, T>` markers and build the selections map type.

### Phase 1.6: Union Identity & Pattern Namespace Ergonomics

_A targeted fix for an oversight in Phases 1 and 1.5: the union factory returned by `union({})` carries no runtime identity linking it to its variants. This means combinators like `is.union(Result, Option)` cannot correctly match a value that is a variant of either union — the factory object is treated as a structural pattern rather than a union membership check. Phase 1.6 closes that gap and takes the opportunity to round out the `is.not.*` namespace for BDD-style pattern authoring._

#### Phase 1.6a — Runtime Union Identity

- **Per-factory unique symbol:** `buildUnion` generates a `const id = Symbol()` on each call. The resulting factory object receives `[unionFactoryTag] = id` as a non-enumerable, symbol-keyed property. Every variant prototype built by that factory receives `[parentUnionId] = id` via `Object.defineProperty` — non-enumerable, symbol-keyed, no footprint on serialization or `Object.keys()`.
- **`evaluatePatternValue` early check:** Before the existing structural-object branch, the pattern evaluator checks `unionFactoryTag in p`. If present, it returns `{}` if `v?.[parentUnionId] === p[unionFactoryTag]` and `false` otherwise. This check is first-position — an efficient early escape for the common case.
- **`is.union(Result, Option)` now works:** Each factory argument is detected via `[unionFactoryTag]` and matched by identity, not structure. `is.not(Result)` and `is.not.variant(Factory)` work as natural side effects.
- **`variantOf` helper:** Export `variantOf(factory, value?)` as a curried standalone — `variantOf(Result)(v)` returns `true` if `v` is a variant of `Result`. Composable with `pred()`: `pred(variantOf(Result))`.
- **`is.variant(factory)`:** Added to the `is` namespace as `pred(variantOf(factory))`. Returns a `Pred` narrowed to the union's instance type (runtime only in 1.6a; precise generic signature lands in 1.6b).

#### Phase 1.6b — Type-Level Narrowing for Union Factories

_Deferred from 1.6a. Runtime correctness ships first; type-level precision follows. Same pattern as 1.0 → 1.5._

- **Phantom brand on the factory type:** The TypeScript return type of `buildUnion` gains a `readonly [unionFactoryTag]: U` where `U` is the union of all variant instance types. This is a type-only declaration — no runtime property is added to variants, and no enumerable key is introduced anywhere.
- **`ExtractNarrowType` extension:** A new branch handles `{ [unionFactoryTag]: infer U } → U`, so `is.union(Result, Option)` narrows the matched field to `Union<typeof Result> | Union<typeof Option>`.
- **`variantOf` and `is.variant` generic signatures:** Both receive their full generic forms, enabling precise narrowing at call sites.

#### Phase 1.6c — `is.not.*` Namespace

_Quality-of-life ergonomics. `is.not(is.string)` is semantically correct but reads backwards for developers who think in BDD terms. `is.not.string` aligns with natural language and mirrors the `is.*` namespace structure._

- **`is.not` remains callable:** `is.not(pattern)` is unchanged. The callable form is the escape hatch for arbitrary patterns — literals, custom `pred(...)`, union factories, other combinators.
- **Pre-computed values (zero-arg wildcards):** Each `is.*` wildcard gets a mirrored counterpart as a plain `NotCombinator` value — not callable, consistent with how `is.string` et al. are plain `Pred` values:
    - `is.not.string`, `is.not.number`, `is.not.boolean`
    - `is.not.nullish`, `is.not.defined`, `is.not.array`, `is.not.object`
- **Parameterized mirrors (functions):** The parameterized members of `is` are mirrored as functions under `is.not`:
    - `is.not.union(...patterns)` — equivalent to `is.not(is.union(...patterns))`
    - `is.not.variant(factory)` — equivalent to `is.not(is.variant(factory))`

### Phase 2: The Data Boundary (The "Schema" Module)

_Aljabr is currently exceptional for internal domain modeling, but needs a bridge to handle untyped external data safely._

- **`aljabr/schema` for Decoding:** Introduce a schema and validation API to transform untyped input (like external JSON from an API) into strictly typed Aljabr union variants. Similar to `effect/Schema`, this API should handle "Decoding" (parsing unknown data into a valid Aljabr variant) and "Encoding" (converting the variant back into a raw format for external systems).
- **Bidirectional Transformation:** A core rule of this roadmap phase is that the schema should guarantee that encoding a decoded variant returns the exact original data format. This closes the gap between untyped forms/APIs and Aljabr's strictly-typed internal state machines.

### Phase 3: Resilient Async Lifecycles & Scheduling

_Enhancing the `aljabr/prelude` to not just track reactive UI state, but actively manage the retry and polling logic required for robust applications._

- **Automated Scheduling Policies:** Integrate a declarative `Schedule` module for `AsyncDerived` that can manage the execution of computations over time. By taking inspiration from Effect, developers should be able to define policies—like exponential backoff with jitter—to dictate exactly how an `AsyncDerived` state automatically retries after entering an `Error` variant.
- **Timeout & Race Primitives:** Build in native timeout handling for reactive effects, allowing `AsyncDerived` states to automatically transition if a computation takes too long, preventing hanging UI operations.

### Phase 4: Resource Scoping and Dependency Injection

_Scaling Aljabr to support complex, enterprise-grade application architectures safely._

- **Resource Lifecycles & Finalizers:** When a reactive effect (like `watchEffect`) opens a WebSocket or a file handle, it must be cleanly disposed of. Implement a `Scope` primitive that attaches "finalizers" to a reactive context, guaranteeing that teardown logic always executes whether the computation succeeds, fails, or is interrupted.
- **Type-Safe Context Injection:** Hardcoded dependencies make unit testing reactive effects difficult. Introduce a dependency injection system where effects can declare requirements via "Tags" (unique identifiers mapping to interfaces). The execution context can then dynamically provide "Layers" (the implementations of those services). This allows developers to easily swap a "Live" database layer for a "Mock" layer during tests.

### Phase 5: Comprehensive Error and Defect Tracking

_Moving beyond just `Result` variants to handle the chaotic reality of the JavaScript runtime._

- **Expected Errors vs. Defects:** While Aljabr's `Result` type perfectly captures known domain errors, unexpected runtime panics (like `TypeError`) can still crash a reactive pipeline. The roadmap should formally differentiate between "Expected Errors" (modeled in the variant types) and "Defects" (unexpected panics).
- **The `Cause` Data Structure:** Introduce a structure to capture the exact outcome of a failed reactive computation. Inspired by Effect TS, this structure should track sequential errors, parallel errors, interruptions, and defects, allowing `AsyncDerived` and `watchEffect` to gracefully trap JavaScript panics instead of crashing the host application.

### **Phase 6: Solid-Inspired Reactive Ergonomics (The Factory API)**

_Focus: Lowering the adoption barrier for frontend engineers by introducing familiar, function-based reactive factories (`createSignal`, `createEffect`, `createMemo`) that seamlessly wrap Aljabr's strict state machines._

You make a fantastic point. If `get()` returns a variant instead of the raw value, we completely destroy the frictionless ergonomics that make Solid's signal pattern so popular in the first place. A developer writing `const [count, setCount] = createSignal(0)` expects `count() + 1` to just work, rather than having to extract the value out of an `Active` variant first.

Here is a revised specification for **Section 6.1** that incorporates your approaches, ensuring we keep the raw value ergonomics while preserving the power of Aljabr's state machines.

---

#### **Revised 6.1: Tuple-Based Signal Initialization (`createSignal`)**

- **The Problem:** Currently, Aljabr signals are instantiated via `Signal.create()` and accessed via class-like object methods. While developers want the familiar `[get, set] = createSignal()` tuple from SolidJS, we cannot simply make `get()` return an Aljabr variant, or we lose the ability to easily read the raw value in UI templates and math operations.
- **The Roadmap Idea:** Introduce a `createSignal(initialValue)` factory that returns a `[getter, setter]` tuple, but supercharge the `getter` using a hybrid function-object approach (or parameter overloads) to serve both use cases natively.

**Implementation Options for the `getter`:**

**Option A: The Property Approach (Recommended)**
Because JavaScript functions are objects, we can attach methods directly to the getter function. Calling the function returns the raw value, while calling a property on the function returns the trackable Aljabr variant.

```typescript
const [count, setCount] = createSignal(0);

// 1. Raw ergonomics: returns the scalar value (e.g., 0)
const raw = count();

// 2. State machine ergonomics: returns the variant (Active(0))
const state = count.state(); // or count.read()

match(count.state(), {
    Active: ({ value }) => console.log(value),
    Unset: () => console.log("No value yet"),
    Disposed: () => console.log("Signal killed"),
});
```

_Why this is strong:_ It is highly self-documenting. `count()` behaves exactly as a Solid developer expects, while `count.state()` provides a clear, discoverable escape hatch into Aljabr's pattern-matching engine. It also maps perfectly to the existing `Signal` API where you might differentiate between getting a raw value and reading the tracked state.

**Option B: The Parameter Overload Approach**
We can use TypeScript function overloads to change the return type based on an optional boolean flag passed to the getter.

```typescript
const [count, setCount] = createSignal(0);

// 1. Raw ergonomics
const raw = count();

// 2. State machine ergonomics
const state = count(true);

match(count(true), { ... });
```

_Why this is strong:_ It keeps the API surface strictly functional without attaching methods to functions. However, passing a "magic boolean" (`true`) is slightly less discoverable via IntelliSense than a named `.state()` property.

**Option C: The "Triple" Tuple Approach**
Instead of overloading the getter, we completely separate the concerns by expanding the returned tuple.

```typescript
const [count, state, setCount] = createSignal(0);

// count() -> returns raw value
// state() -> returns Aljabr variant
```

_Why this is strong:_ It is explicit and prevents the need to attach properties to functions. However, it diverges slightly from the exact `[get, set]` muscle memory developers have from React and Solid.

#### **6.2 Idiomatic Computed Values (`createMemo` & `createResource`)**

- **The Problem:** Aljabr uses `Derived.create()` for synchronous lazy computations and `AsyncDerived.create()` for async state. The naming convention diverges from popular reactive frameworks.
- **The Roadmap Idea:**
    - Map `createMemo` as an ergonomic alias/wrapper for `Derived.create()`, returning a simple getter function.
    - Introduce `createResource` (mirroring Solid's async handling) as a wrapper around `AsyncDerived<T, E>`. Because `AsyncDerived` natively exposes a `Reloading` and `Stale` state, it inherently solves the "stale-while-revalidate" pattern that Solid developers often have to manage manually.

#### **6.3 Unified Effect Execution (`createEffect`)**

- **The Problem:** Aljabr currently provides `watchEffect` to run asynchronous thunks reactively.
- **The Roadmap Idea:** Introduce `createEffect` as the standard, familiar factory. It will automatically track any signals read within its execution block and re-run when they change. Crucially, `createEffect` should natively return Aljabr's `Disposed` variant when the component/context unmounts, guaranteeing safe cleanup of side-effects.

#### **6.4 Scoped Reactivity (`createRoot` / `createOwner`)**

- **The Problem:** Reactivity needs boundaries to prevent memory leaks when components unmount.
- **The Roadmap Idea:** Aljabr already has `createOwner` and `runInContext` inside its prelude. Formalize a `createRoot` factory that aligns with Solid's disposable reactive boundaries, allowing developers to safely nest and destroy isolated signal graphs without leaking memory.

### **7.0 Phase 7: Ecosystem Bindings & UI Adapters (Strategy Body)**

**7.1 External State Hooks (`@aljabr/react`)**
The primary goal of the React adapter is to allow React components to subscribe to Aljabr's reactive primitives (`Signal`, `Derived`, `Ref`) without delegating the actual state management to React.

- **Implementation:** These hooks (e.g., `useSignal`, `useDerived`, `useRefPath`) will be strictly powered by React 18's `useSyncExternalStore`. This guarantees tear-free, concurrent-safe rendering while ensuring that Aljabr remains the single, uncompromising source of truth for the application's state machine.

**7.2 Form Resolvers (`@aljabr/hookform-resolver`)**
To seamlessly connect Aljabr's data validation (from Phase 2) with industry-standard UI practices, Aljabr will provide official resolvers for libraries like React Hook Form.

- **Implementation:** The resolver will act as a translation layer. It will take untyped form inputs, pass them through an Aljabr schema, and use Aljabr's native `match()` to evaluate the `Result` or `Validation` container. Domain errors will be accumulated and mapped precisely to the nested field formats expected by the UI form libraries.

**7.3 "Roll-Your-Own" UI Matching**
Aljabr will intentionally **not** ship custom React components (like `<Match>` or `<Switch>`) to handle conditional UI rendering.

- **Rationale:** Providing UI components bloats the API surface, ties the library too closely to specific framework paradigms (e.g., children props, context), and is ultimately unnecessary. Because Aljabr already exports a highly capable and strictly-typed `match()` function, developers who desire a declarative `<Match>` component in JSX can trivially wrap the native `match()` engine and "roll their own" in just a few lines of code. This keeps Aljabr's footprint minimal while giving users the freedom to construct their view layers exactly how they prefer.
