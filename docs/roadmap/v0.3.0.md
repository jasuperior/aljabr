### **Aljabr Future Roadmap (v0.3.0): Formal Overview**

**Strategic Vision**

Aljabr currently provides an exceptionally tight, zero-dependency pipeline for defining Algebraic Data Types (ADTs), utilizing non-enumerable symbol tags for clean serialization, and mapping these domain models directly to reactive UI state via the `aljabr/prelude`. While it excels at enforcing compile-time exhaustiveness and eliminating boilerplate for internal data models, as applications scale, developers encounter friction at the boundaries of their domain (e.g., untyped API payloads) and within complex, long-running asynchronous effects.

This roadmap outlines a progressive, six-phase expansion for Aljabr. By drawing inspiration from best-in-class libraries like `ts-pattern` for structural ergonomics, `@effect/schema` for data validation, the `Effect` ecosystem for enterprise-grade resource and defect management, and `SolidJS` for reactive ergonomics, Aljabr can bridge the gap between frontend state machines and robust, fault-tolerant application architectures. The goal is to retain Aljabr's lightweight, tag-first philosophy while delivering the primitives necessary for deep structural extraction, bidirectional data validation, dependency injection, resilient async lifecycles, and familiar reactive workflows.

### Phase 1: Deep Structural Matching & Ergonomics

_The immediate goal is to make Aljabr's `when()` arms as expressive as best-in-class pattern matchers, without losing the performance benefits of prototype-bound symbol tags._

- **Deep Structural Extraction (`select` bindings):** Currently, pulling nested data out of an Aljabr variant requires manual destructuring in the handler. The roadmap should introduce an extraction utility akin to `ts-pattern`'s `P.select()`, which allows developers to extract a specific nested property directly within the pattern and inject it as a typed argument into the handler.
- **Wildcards and Combinators:** Expand the pattern matching API to support robust combinators such as `P.union`, `P.intersection`, and `P.not`. Adding primitives like `P.string` or `P.number` will allow developers to match variants based on both identity and the shape of their internal payloads simultaneously.
- **Align with TC39 Pattern Matching:** As the ECMAScript Pattern Matching proposal advances, ensure Aljabr's matcher is positioned to smoothly interoperate with or compile down to native `match` expressions and "extractor patterns".

### Phase 1.5: Type-Level Inference for Combinators and `select`

_Deferred from Phase 1. Runtime behavior ships first; type-level precision follows._

- **Combinator Narrowing:** `is.not(is.string)` should narrow the matched value's type to `Exclude<T, string>`. `is.union(is.string, is.number)` should narrow to `string | number`. This requires type-level recursion over combinator arguments to compute the inferred type. Phase 1 ships these with `unknown` as the narrowed type; Phase 1.5 promotes them to precise conditional-type inference.
- **`select` Inference:** Given a pattern containing `select("name")` at any depth, the `selections` second argument to the handler should be typed as `{ name: FieldType }` rather than `Record<string, unknown>`. This requires type-level traversal of the pattern type to collect all `Select<Name, T>` markers and build the selections map type.

### Phase 1.6: Union Identity & Pattern Namespace Ergonomics

_A targeted fix for an oversight in Phases 1 and 1.5: the union factory returned by `union({})` carries no runtime identity linking it to its variants. This means combinators like `is.union(Result, Option)` cannot correctly match a value that is a variant of either union — the factory object is treated as a structural pattern rather than a union membership check. Phase 1.6 closes that gap and takes the opportunity to round out the `is.not.*` namespace for BDD-style pattern authoring._

#### Phase 1.6a — Runtime Union Identity

- **Per-factory unique symbol:** `buildUnion` generates a `const id = Symbol()` on each call. The resulting factory object receives `[unionFactoryTag] = id` as a non-enumerable, symbol-keyed property. Every variant prototype built by that factory receives `[parentUnionId] = id` via `Object.defineProperty` — non-enumerable, symbol-keyed, no footprint on serialization or `Object.keys()`.
- **`evaluatePatternValue` early check:** Before the existing structural-object branch, the pattern evaluator checks `unionFactoryTag in p`. If present, it returns `{}` if `v?.[parentUnionId] === p[unionFactoryTag]` and `false` otherwise. This check is first-position — an efficient early escape for the common case.
- **`is.union(Result, Option)` now works:** Each factory argument is detected via `[unionFactoryTag]` and matched by identity, not structure. `is.not(Result)` and `is.not.variant(Factory)` work as natural side effects.
- **`variantOf` helper:** Export `variantOf(factory, value?)` as a curried standalone — `variantOf(Result)(v)` returns `true` if `v` is a variant of `Result`. Composable with `pred()`: `pred(variantOf(Result))`.
- **`is.variant(factory)`:** Added to the `is` namespace as `pred(variantOf(factory))`. Returns a `Pred` narrowed to the union's instance type (runtime only in 1.6a; precise generic signature lands in 1.6b).

#### Phase 1.6b — Type-Level Narrowing for Union Factories

_Deferred from 1.6a. Runtime correctness ships first; type-level precision follows. Same pattern as 1.0 → 1.5._

- **Phantom brand on the factory type:** The TypeScript return type of `buildUnion` gains a `readonly [unionFactoryTag]: U` where `U` is the union of all variant instance types. This is a type-only declaration — no runtime property is added to variants, and no enumerable key is introduced anywhere.
- **`ExtractNarrowType` extension:** A new branch handles `{ [unionFactoryTag]: infer U } → U`, so `is.union(Result, Option)` narrows the matched field to `Union<typeof Result> | Union<typeof Option>`.
- **`variantOf` and `is.variant` generic signatures:** Both receive their full generic forms, enabling precise narrowing at call sites.

#### Phase 1.6c — `is.not.*` Namespace

_Quality-of-life ergonomics. `is.not(is.string)` is semantically correct but reads backwards for developers who think in BDD terms. `is.not.string` aligns with natural language and mirrors the `is.*` namespace structure._

- **`is.not` remains callable:** `is.not(pattern)` is unchanged. The callable form is the escape hatch for arbitrary patterns — literals, custom `pred(...)`, union factories, other combinators.
- **Pre-computed values (zero-arg wildcards):** Each `is.*` wildcard gets a mirrored counterpart as a plain `NotCombinator` value — not callable, consistent with how `is.string` et al. are plain `Pred` values:
    - `is.not.string`, `is.not.number`, `is.not.boolean`
    - `is.not.nullish`, `is.not.defined`, `is.not.array`, `is.not.object`
- **Parameterized mirrors (functions):** The parameterized members of `is` are mirrored as functions under `is.not`:
    - `is.not.union(...patterns)` — equivalent to `is.not(is.union(...patterns))`
    - `is.not.variant(factory)` — equivalent to `is.not(is.variant(factory))`

### Phase 2: The Data Boundary (The "Schema" Module)

_Aljabr is currently exceptional for internal domain modeling, but needs a bridge to handle untyped external data safely. The goal of this phase is to migrate the external world (untyped API payloads, form inputs, raw JSON) into the internal Aljabr world of typed union variants — using the same match engine and ADT patterns developers already know._

#### Pre-requisite: `Validation` fixes

Before Phase 2a ships, two targeted fixes to `Validation` must land:

- **Flatten `Combinable.combine()`:** Currently `Valid(a).combine(Valid(b)).combine(Valid(c))` produces `Valid([[a, b], c])`. The fix introduces a `CombineValues<A, B>` conditional type (`A extends readonly unknown[] ? [...A, B] : [A, B]`) and a runtime spread so chaining always produces a flat tuple. Invalid error accumulation is already flat and unchanged.
- **`Validation.all([...])`:** A static method accepting a fixed-length tuple of `Validation` values and returning a single `Validation` whose `Valid` payload is a typed tuple of all values. Idiomatic for the multi-field validation pattern the schema engine depends on.

#### Phase 2a — Schema Union & Decode

_Runtime decode behavior ships first. Encode and the adapter layer follow in 2b._

**Architecture — Option C (native engine + adapter layer):**
The module ships a self-contained, zero-dependency schema DSL built on Aljabr's own union system. An adapter layer (for wrapping Zod, ArkType, etc.) follows in Phase 2b. The primary path is the native engine; adapters are the escape hatch.

**Entry point — `aljabr/schema`:**
A separate bundle entry point (alongside `aljabr/prelude`), so users doing pure state-machine work pay no bundle cost for the schema machinery.

**`Schema<T>` is an Aljabr union:**
Schemas are first-class introspectable variants — not plain objects or classes. This means you can `match` over a schema, traverse its structure, and eventually derive JSON Schema or OpenAPI specs from the same value you use to decode. The variant set for Phase 2a:

| Schema variant      | Output type                |
| ------------------- | -------------------------- |
| `StringSchema`      | `string`                   |
| `NumberSchema`      | `number`                   |
| `BooleanSchema`     | `boolean`                  |
| `LiteralSchema<V>`  | `V`                        |
| `OptionalSchema<T>` | `T \| undefined`           |
| `NullableSchema<T>` | `T \| null`                |
| `NullishSchema<T>`  | `T \| null \| undefined`   |
| `ArraySchema<T>`    | `T[]`                      |
| `ObjectSchema<T>`   | `{ [K in keyof T]: T[K] }` |
| `UnionSchema<Ts>`   | `Ts[number]`               |
| `VariantSchema<F>`  | `Union<typeof F>`          |

**Decode output — `Validation<T, DecodeError>`:**
`decode(schema, unknown)` returns `Validation<T, DecodeError>`. Because decoding unknown data can have _multiple_ simultaneous field errors (missing field here, type mismatch there), the `Validation` union is the correct container — not `Result`.

**`DecodeError` is a tagged union:**
All variants carry a `path: (string | number)[]` payload tracing where in the input the error occurred, accumulated recursively as the decoder descends into objects and arrays.

| Variant               | Additional payload                                      |
| --------------------- | ------------------------------------------------------- |
| `TypeMismatch`        | `expected: string, received: string`                    |
| `MissingField`        | `field: string`                                         |
| `InvalidLiteral`      | `expected: unknown, received: unknown`                  |
| `UnrecognizedVariant` | `discriminant: string, received: string`                |
| `Custom`              | `message: string` — escape hatch for future `.refine()` |

**`Schema.variant(Factory, shapeMap, options?)`:**
The bridge primitive that decodes external data directly into an Aljabr union variant — the core of "migrating the external world to the internal world." Options:

- `discriminant` (default `"type"`): the key on the raw input that identifies which variant to produce.
- `map`: optional record mapping external discriminant values to Aljabr variant names (e.g. `{ ok: "Accept", err: "Reject" }`). Without a map, the discriminant value must match the variant name exactly.

**`Schema.object(shape, options?)`:**
Options accept `mode: "strip" | "strict" | "passthrough"` (default `"strip"`). Surfacing `"strip"` explicitly allows developers to self-document the intended behavior even when relying on the default.

**`Schema.union(...schemas)`:**
Tries each schema left-to-right at decode time; returns the first `Valid` result. For discriminated external shapes, `Schema.variant` is the idiomatic path — `Schema.union` handles plain type-level unions like `string | number`.

#### Phase 2b — Encode & Codec Adapter Layer

_Deferred from 2a. Decode is the 90% use case; encode and third-party adapters are additive._

**Two distinct types — `Decoder<I, O>` and `Codec<I, O>`:**
Rather than a single interface with an optional `encode`, Phase 2b introduces two named types that reflect genuinely different capabilities:

```ts
interface Decoder<I, O> {
    decode(input: I): Validation<O, DecodeError>;
}
interface Codec<I, O> extends Decoder<I, O> {
    encode(output: O): I;
}
```

Functions that only decode accept `Decoder<I, O>`. Functions that require round-tripping declare `Codec<I, O>`. No runtime narrowing needed — the type system enforces the distinction at call sites.

**Encode is total:** `encode(O): I` never fails. If a value decoded successfully, encoding it back is unconditional by design — that's the round-trip guarantee. Fallible encode is not supported; `defineCodec` gives escape-hatch power to anyone who needs custom behavior.

**`Schema.variant` encode — full bidirectional map:** If a `map` config was provided during schema construction (`{ ok: "Accept" }`), encode inverts it (`Accept` → `"ok"`). The round-trip guarantee holds regardless of discriminant configuration.

**Encode on all native schemas:** Each Phase 2a schema variant gains an `encode` implementation, making every native schema a full `Codec<unknown, T>` by default.

**`Schema.transform`:** Ships in 2b as the bridge between `Decoder` and `Codec`. Takes any `Decoder<I, O>`, an encode function `(P) => I`, and produces a `Codec<I, P>`. The canonical use case is type coercion at boundaries:

```ts
const DateSchema = Schema.transform(
    Schema.string(),
    (s) => new Date(s), // decode: string → Date
    (d) => d.toISOString(), // encode: Date → string
);
// → Codec<unknown, Date>
```

**Adapter protocol — `defineCodec` and `defineDecoder`:** Zero-cost identity helpers (no runtime behavior) that infer `I` and `O` from provided functions. Adapter authors get type errors at definition time rather than at call sites:

```ts
const myDecoder = defineDecoder({ decode: (input) => ... })
const myCodec   = defineCodec({ decode: (input) => ..., encode: (output) => ... })
```

**No official adapter packages:** Aljabr ships zero runtime dependencies. Official `@aljabr/zod` or `@aljabr/arktype` packages will never exist. Instead, the docs ship two complete, copy/paste-ready adapter examples (one for Zod, one for ArkType) covering the two most widely-used validators. Developers take ownership of the bridge in their own codebase.

**`roundtrip` test utility:** A dev/test helper exported from `aljabr/schema` that runs `encode(decode(input))` and performs a deep equality check against the original input. Codec authors call it in their test suites to verify correctness. No production overhead.

### Phase 3: Resilient Async Lifecycles & Scheduling

_Hardening `AsyncDerived` and `watchEffect` against the realities of network and async UI state — retry, backoff, timeout, and cancellation — without requiring a separate entry point. All primitives ship inside `aljabr/prelude`._

#### `Schedule` Union

A declarative retry-delay policy, modeled as an Aljabr union. The scheduler reads a `Schedule` value and computes the next delay after each failed attempt.

| Variant       | Payload                                                                             |
| ------------- | ----------------------------------------------------------------------------------- |
| `Fixed`       | `{ delay: number }`                                                                 |
| `Linear`      | `{ delayPerAttempt: number; jitter?: boolean }`                                     |
| `Exponential` | `{ initialDelay: number; maxDelay: number; multiplier?: number; jitter?: boolean }` |
| `Custom`      | `{ fn: (attempt: number, error: unknown) => number \| null }`                       |

`Custom.fn` returning `null` signals unconditional termination regardless of `maxRetries`.

#### `ScheduleError` Union

Errors produced by the scheduler itself — distinct from domain errors thrown by the thunk. Fully matchable.

| Variant              | Payload                                    |
| -------------------- | ------------------------------------------ |
| `TimedOut`           | `{ elapsed: number; timeout: number }`     |
| `MaxRetriesExceeded` | `{ attempts: number; lastError: unknown }` |

#### `AsyncOptions<E>` — Shared Options Bag

Applied identically to `AsyncDerived.create()` and `watchEffect()`:

```ts
type AsyncOptions<E> = {
    schedule?: Schedule;
    maxRetries?: number; // undefined = retry indefinitely
    shouldRetry?: (error: E) => boolean; // default: always true
    timeout?: number; // ms — undefined = no timeout
    onRetry?: (attempt: number, error: E, nextDelay: number) => void;
};
```

#### `AbortSignal` Threading

The thunk for both `AsyncDerived` and `watchEffect` receives an `AbortSignal` as its first argument. The scheduler aborts the previous signal before each new attempt, preventing stale in-flight requests from resolving into current state.

```ts
AsyncDerived.create((signal) => fetch("/api/data", { signal }), options);
watchEffect((signal) => fetch("/api/data", { signal }), options);
```

#### State Machine Changes

**`AsyncFailed<E>`** gains a richer payload to expose retry context at read time:

```ts
AsyncFailed<E>: { error: E; attempts: number; nextRetryAt: number | null }
```

`nextRetryAt` is a millisecond timestamp of the next scheduled attempt, or `null` when the machine has given up (max retries exceeded, `shouldRetry` returned false, or `Custom.fn` returned `null`).

**`Effect`** gains a `Failed<E>` variant with the same payload, passed to `onChange` alongside `Done` and `Stale`. Errors are removed from `Done`, which now represents successful settlement only. `onChange` fires on `Done`, `Stale`, and `Failed` — not on `Idle` or `Running`.

#### Timeout Semantics

A timeout fires as `Failed(ScheduleError.TimedOut(...))` and counts as one attempt toward `maxRetries`. Users who want timeouts to be immediately terminal pass `shouldRetry: (e) => !match(e, { TimedOut: () => true, ... })`.

#### Polling

Time-driven polling (interval-based re-evaluation independent of dependency changes) is deferred to Phase 4, where it fits naturally within the `Scope` and finalizer model.

### Phase 4: Resource Scoping

_Giving `watchEffect` and `AsyncDerived` structural resource safety — guaranteed teardown on success, failure, or interruption._

#### `Scope`

A `Scope` is a new public abstraction over the internal `Computation` owner node. It does not expose reactive machinery (`dirty`, `sources`, subscribers) — only resource lifetime primitives. `Scope` wraps a `Computation` internally and is the official public API for resource lifetime management.

**Creation:** `Scope()` is a callable factory (consistent with `Signal`, `Resource`, etc.). Called with no arguments it auto-parents to `getCurrentComputation()`, inheriting the current reactive owner. Calling it at the top level (no active computation) creates a root scope.

**API surface:**

```ts
interface ScopeHandle {
    readonly state: Active | Disposed; // non-reactive union snapshot, extensible
    defer(fn: () => Promise<void> | void): void;
    acquire<T>(resource: Resource<T>): Promise<T>;
    dispose(): Promise<void>;
    [Symbol.asyncDispose](): Promise<void>; // alias for dispose()
}
```

- **`scope.defer(fn)`** — registers a finalizer. Finalizers run in **LIFO order** on disposal. If `fn` returns a `Promise`, disposal awaits it before moving to the next finalizer. A rejected finalizer logs a warning and the chain continues (best-effort — Phase 5 will harden this with defect tracking).
- **`scope.acquire(resource)`** — acquires the resource (calls `resource.acquire()`), registers `resource.release(value)` as a `defer` finalizer, and returns `Promise<T>`.
- **`scope.dispose()`** — runs all finalizers in LIFO order, disposes all child scopes, returns `Promise<void>`.
- **`scope.state`** — a plain (non-reactive) union snapshot. Two variants for Phase 4: `Active` and `Disposed`. Not tracked; read it explicitly to inspect. Designed for extensibility (`Errored`, `Suspended`, etc. can be added in future phases without changing the API shape).
- **`Symbol.asyncDispose`** — enables `await using scope = Scope()` via the TC39 Explicit Resource Management proposal. Purely additive — an alias for `dispose()`.

**Disposal cascade:** When a parent scope (or the owning `Computation`) disposes, all child scopes dispose automatically before the parent's own finalizers run. Structural safety is guaranteed.

**`onDispose` is replaced:** `defer` is the single finalizer primitive across both the explicit (`scope.defer`) and implicit path (see below). There is no separate `onDispose`.

#### Implicit Hooks

Inside a `watchEffect` or `AsyncDerived` thunk, two standalone functions are available that operate on the thunk's own `Scope`:

```ts
defer(fn); // registers a finalizer on the current scope
acquire(resource); // acquires a resource via the current scope → Promise<T>
```

Both throw if called outside an active computation (no current scope on the stack). This is intentional — silent no-ops would hide misuse.

#### `watchEffect` and `AsyncDerived` thunk signature

Both thunks receive an explicit `Scope` argument alongside `AbortSignal`:

```ts
watchEffect(
    (signal, scope) => {
        const db = await scope.acquire(DbResource);
        return fetch("/api", { signal });
    },
    onChange,
    options,
);

AsyncDerived.create((signal, scope) => {
    defer(() => cleanup()); // implicit path also available
    return fetch("/api", { signal }).then((r) => r.json());
}, options);
```

The `Scope` argument enables explicit resource management and introspection. The implicit `defer` / `acquire` functions are available as a convenience for simple cases.

#### `Resource`

`Resource(acquire, release)` is a callable factory pairing an async acquisition with a release:

```ts
interface ResourceHandle<T> {
    acquire: () => Promise<T>;
    release: (value: T) => Promise<void> | void;
}

const DbResource = Resource(
    () => connectToDb(url), // async acquire
    (db) => db.disconnect(), // sync or async release
);
```

Resources are inert until consumed via `scope.acquire(resource)` or the implicit `acquire(resource)`. Acquiring registers release as a `scope.defer` finalizer automatically — the resource's lifetime is structurally tied to the scope.

**Error handling (Phase 4 — naive):** A rejected `acquire` propagates as a rejected `Promise`. A rejected `release` is caught, warned, and skipped (best-effort). Phase 5's defect tracking will harden both paths.

#### Deferred: Context / `createRoot`

Type-safe context threading (originally described as Tag + Layer dependency injection) is **deferred to Phase 6**. The Tag/Layer model was found to be too similar to Effect-TS and sits outside Aljabr's design philosophy. The correct primitive — a `Context` factory for threading values down the owner tree — belongs alongside `createRoot` in Phase 6, where reactive scoping ergonomics are the focus. Both `Context` and `createRoot` serve the same structural goal and should be designed together.

### Phase 5: Comprehensive Error and Defect Tracking

_Moving beyond just `Result` variants to handle the chaotic reality of the JavaScript runtime._

- **Expected Errors vs. Defects:** While Aljabr's `Result` type perfectly captures known domain errors, unexpected runtime panics (like `TypeError`) can still crash a reactive pipeline. The roadmap should formally differentiate between "Expected Errors" (modeled in the variant types) and "Defects" (unexpected panics).
- **The `Cause` Data Structure:** Introduce a structure to capture the exact outcome of a failed reactive computation. Inspired by Effect TS, this structure should track sequential errors, parallel errors, interruptions, and defects, allowing `AsyncDerived` and `watchEffect` to gracefully trap JavaScript panics instead of crashing the host application.

### **Phase 6: Solid-Inspired Reactive Ergonomics (The Factory API)**

_Lowering the adoption barrier for frontend engineers by introducing familiar, function-based reactive factories (`createSignal`, `createEffect`, `createMemo`) that seamlessly wrap Aljabr's strict state machines while preserving raw value ergonomics._

#### 6.1 Tuple-Based Signal Initialization (`createSignal`)

`createSignal(initialValue)` returns a `[getter, setter]` tuple. The getter must serve two audiences simultaneously — raw value reads for UI templates and math, and variant reads for pattern-matching — without forcing callers to choose a path at construction time.

**Preferred design — property-augmented getter:**
Because JavaScript functions are objects, methods can be attached directly to the getter. Calling the function returns the raw scalar; calling `.state()` returns the trackable Aljabr variant.

```typescript
const [count, setCount] = createSignal(0);

count(); // → 0         (raw value)
count.state(); // → Active(0) (Aljabr variant)

match(count.state(), {
    Active: ({ value }) => console.log(value),
    Unset: () => console.log("No value yet"),
    Disposed: () => console.log("Signal killed"),
});
```

**Alternative design — overloaded getter:**
TypeScript function overloads can change the return type based on an optional flag, keeping the API surface strictly functional at the cost of a less-discoverable escape hatch.

```typescript
count(); // → 0           (raw value)
count(true); // → Active(0)   (Aljabr variant)
```

#### 6.2 Idiomatic Computed Values (`createMemo` & `createResource`)

- **`createMemo`:** An ergonomic alias for `Derived.create()`, returning a simple getter function. Naming aligns with the SolidJS convention developers already know.
- **`createResource`:** A wrapper around `AsyncDerived<T, E>` mirroring Solid's async primitive. Because `AsyncDerived` natively exposes `Reloading` and `Stale` states, the stale-while-revalidate pattern is handled without additional configuration.

#### 6.3 Unified Effect Execution (`createEffect`)

Introduce `createEffect` as the canonical effect factory, replacing the lower-level `watchEffect` for most use cases. It automatically tracks signals read within its execution block and re-runs when they change. When the owning context unmounts, `createEffect` transitions to Aljabr's `Disposed` variant, guaranteeing deterministic cleanup of side-effects.

#### 6.4 Scoped Reactivity (`createRoot`) and Context Threading (`Context`)

Formalize a `createRoot` factory over the existing `createOwner` / `runInContext` internals. `createRoot` establishes a disposable reactive boundary aligned with Solid's model, allowing developers to safely nest and destroy isolated signal graphs without leaking memory.

`Context` is designed alongside `createRoot` as they share the same structural goal. A `Context(defaultValue)` factory threads an arbitrary value down the owner tree — any descendant computation can read the nearest provided value via `Context.use()`, falling back to the default if none is found. This is distinct from `Scope` (which manages resource lifetime) and from the owner tree itself (which manages reactive tracking). `Context` is the idiomatic Aljabr replacement for the Tag/Layer dependency injection pattern — lighter, reactive-native, and consistent with the library's factory-first philosophy.

### **7.0 Phase 7: Ecosystem Bindings & UI Adapters (Strategy Body)**

**7.1 External State Hooks (`@aljabr/react`)**
The primary goal of the React adapter is to allow React components to subscribe to Aljabr's reactive primitives (`Signal`, `Derived`, `Ref`) without delegating the actual state management to React.

- **Implementation:** These hooks (e.g., `useSignal`, `useDerived`, `useRefPath`) will be strictly powered by React 18's `useSyncExternalStore`. This guarantees tear-free, concurrent-safe rendering while ensuring that Aljabr remains the single, uncompromising source of truth for the application's state machine.

**7.2 Form Resolvers (`@aljabr/hookform-resolver`)**
To seamlessly connect Aljabr's data validation (from Phase 2) with industry-standard UI practices, Aljabr will provide official resolvers for libraries like React Hook Form.

- **Implementation:** The resolver will act as a translation layer. It will take untyped form inputs, pass them through an Aljabr schema, and use Aljabr's native `match()` to evaluate the `Result` or `Validation` container. Domain errors will be accumulated and mapped precisely to the nested field formats expected by the UI form libraries.

**7.3 "Roll-Your-Own" UI Matching**
Aljabr will intentionally **not** ship custom React components (like `<Match>` or `<Switch>`) to handle conditional UI rendering.

- **Rationale:** Providing UI components bloats the API surface, ties the library too closely to specific framework paradigms (e.g., children props, context), and is ultimately unnecessary. Because Aljabr already exports a highly capable and strictly-typed `match()` function, developers who desire a declarative `<Match>` component in JSX can trivially wrap the native `match()` engine and "roll their own" in just a few lines of code. This keeps Aljabr's footprint minimal while giving users the freedom to construct their view layers exactly how they prefer.
