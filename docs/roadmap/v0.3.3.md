### **Aljabr Roadmap (v0.3.1): Ecosystem Bindings & UI Adapters**

_Patch release following v0.3.0. Focused entirely on bridging Aljabr's reactive primitives and schema validation into the React ecosystem._

---

### **Open Question: Library vs. Docs**

Before implementation begins, the delivery format for each adapter needs to be settled. The options are:

- **Publishable package** ŌĆö ships as `@aljabr/react` or `@aljabr/hookform-resolver` on npm, versioned alongside the core
- **Docs example** ŌĆö a complete, copy-paste-ready implementation in the documentation (same pattern as the Zod/ArkType codec adapters in Phase 2b)
- **Mix** ŌĆö publish the high-value, high-complexity adapter as a package; document the simpler one as a copy-paste example

The guiding principle: if the implementation is small and unlikely to require versioned updates in lockstep with the core, a well-written docs example serves users just as well as a published package ŌĆö with zero maintenance overhead.

Current leaning per adapter:

| Adapter | Leaning | Rationale |
|---|---|---|
| `@aljabr/react` | **Package** | `useSyncExternalStore` wiring is non-trivial; users should not have to get concurrent-mode subtleties right themselves |
| `@aljabr/hookform-resolver` | **Docs example** | ~50 lines, no moving parts, users who need it will customize it anyway |

_This is not settled. Revisit before implementation._

### **Repo Structure**

Both adapters (if published) live as npm workspace packages under `adapters/`. The core `aljabr` package is untouched ŌĆö no renamed entry points, no new dependencies.

```
aljabr/               ŌåÉ core, published as "aljabr" (unchanged)
adapters/
  react/              ŌåÉ published as "@aljabr/react"
  hookform-resolver/  ŌåÉ published as "@aljabr/hookform-resolver" (if packaged)
```

The only change to the core `package.json` is adding `"workspaces": ["adapters/*"]`. npm workspaces (v7+) ŌĆö no pnpm required.

### **Phase 1: `@aljabr/react`**

Allows React components to subscribe to Aljabr reactive primitives via React 18's `useSyncExternalStore`, guaranteeing tear-free, concurrent-safe rendering. Aljabr remains the single source of truth; React is purely a subscriber.

**`useSignal` ŌĆö two calling forms:**

```ts
// Low-level: subscribe to an existing Signal<T> (shared/global state)
const value = useSignal(mySig);

// High-level: create + subscribe in one shot (component-local reactive state)
const [count, setCount] = useSignal(0);
```

The low-level form is the primary value-add ŌĆö state that lives outside the component tree (module scope, stores, owner trees) drives React renders without lifting into React's own state model. The high-level form is a convenience wrapper.

**`useDerived` ŌĆö subscribe to a `Derived<T>` or `memo()` getter:**

```ts
const doubled = useDerived(myDerived); // raw T | null, re-renders on change
```

**`useRefPath` ŌĆö not included.** Users who need to subscribe to a path inside a `Ref` pass `ref.get("path")` directly to `useSignal`.

**No JSX components.** `<Match>`, `<Switch>`, and similar declarative rendering components are intentionally out of scope. `match()` is already a function ŌĆö users who want JSX ergonomics can wrap it themselves in a few lines.

### **Phase 2: `@aljabr/hookform-resolver` (or Docs Example)**

A translation layer between Aljabr's schema validation (`aljabr/schema`, Phase 2 of v0.3.0) and React Hook Form's resolver protocol. Takes untyped form inputs, runs them through an Aljabr `Schema`, and maps `DecodeError` paths to React Hook Form's nested field error format.

```ts
import { ajlabrResolver } from "@aljabr/hookform-resolver";

const { register, handleSubmit } = useForm({
    resolver: ajlabrResolver(MySchema),
});
```

`DecodeError` variants (`MissingField`, `TypeMismatch`, `InvalidLiteral`, etc.) are matched exhaustively and mapped to `{ type, message }` entries keyed by their `path` array ŌĆö the format React Hook Form expects.
