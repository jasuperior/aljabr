# Aljabr Roadmap (v0.3.3): Native UI Layer — Reference Implementation

_Minor release following v0.3.2. Introduces `aljabr/ui` and `aljabr/ui/dom`: a signals-driven, pluggable UI rendering layer built natively on Aljabr's reactive primitives. This release ships the core architecture and a working DOM reference implementation. Polish and advanced renderer features are deferred to v0.3.4._

---

## Motivation

The original v0.3.3 plan was a React adapter layer (`@aljabr/react`). After review, a custom UI layer is the more coherent direction: Aljabr's reactive primitives (`Signal`, `Derived`, `Ref`, `ReactiveArray`) already provide everything a rendering layer needs. An adapter would make React the rendering authority; a native layer makes Aljabr the rendering authority, with other targets (canvas, SSR, terminal) as equal peers.

The `@aljabr/react` adapter remains potentially valid for users who want React interop without adopting the Aljabr UI model — that decision is deferred to contributors.

---

## Design Decisions

### Pluggable renderer via `createRenderer`

`createRenderer(host)` returns a `view` factory and `mount` function bound to a specific host implementation. No global renderer state. Multiple renderers can coexist in the same application.

```ts
import { createRenderer } from "aljabr/ui";
import { domHost } from "aljabr/ui/dom";

const { view, mount } = createRenderer(domHost);

mount(() => view("div", { class: "app" }, "hello"), document.body);
```

The `RendererHost<Node, Element>` interface is the contract every host must implement:

```ts
interface RendererHost<Node, Element extends Node> {
    createElement(tag: string): Element;
    createText(text: string): Node;
    insert(parent: Element, child: Node, anchor?: Node): void;
    remove(child: Node): void;
    setProperty(el: Element, key: string, value: unknown): void;
    setText(node: Node, text: string): void;
    parentNode(node: Node): Element | null;
    nextSibling(node: Node): Node | null;
    onMount?(el: Element): void;
    onUnmount?(el: Element): void;
    onUpdate?(el: Element): void;
}
```

Custom renderer targets (canvas, SSR, terminal) implement this interface with different `Node` and `Element` type parameters. The DOM host is the default implementation shipped in `aljabr/ui/dom`. An SSR host is a natural consequence of the pluggable design — no special accommodation required.

---

### `RendererProtocol` — batching escape hatch

Mirrors the `signal` / `signal.protocol` pattern. The simple path (no protocol) is sufficient for most renderers. Targets that need to control when updates flush (e.g., canvas renderers batching into `requestAnimationFrame`) pass a protocol as a second argument:

```ts
const { view, mount } = createRenderer(canvasHost, {
    scheduleFlush(flush) {
        requestAnimationFrame(flush);
    },
});
```

The interface is defined in v0.3.3. The DOM host does not use it yet — full implementation is deferred to v0.3.4.

```ts
interface RendererProtocol {
    scheduleFlush(flush: () => void): void;
}
```

---

### `ViewNode` union — typed intermediate representation

`view()` does not return raw host nodes. It returns a `ViewNode` tagged union, which the renderer immediately consumes (eager, not deferred — no VDOM diffing loop). The union gives the reconciler exhaustive, compile-time-safe dispatch via `match()` instead of fragile `typeof`/`instanceof` guards.

```ts
const ViewNode = union({
    Element: (p: {
        tag: string;
        props: Record<string, unknown>;
        children: ViewNode[];
    }) => p,
    Text: (content: string) => ({ content }),
    Component: (p: {
        fn: (props: Record<string, unknown>) => ViewNode;
        props: Record<string, unknown>;
    }) => p,
    Fragment: (children: ViewNode[]) => ({ children }),
});
```

Variants are consumed synchronously on creation. There is no accumulation step, no tree diffing, and no reconciler loop. Reactivity is handled by the existing signal layer — the `ViewNode` union is a typed dispatch mechanism, not a shadow DOM.

---

### `view()` — the factory function

`view()` is the primary authoring primitive. It is also the JSX factory target.

```ts
// Direct usage
view('button', { onClick: () => store.submit() }, label)

// JSX — requires tsconfig configuration
<button onClick={() => store.submit()}>{label()}</button>
```

JSX configuration:

```json
// tsconfig.json
{ "compilerOptions": { "jsxImportSource": "aljabr/ui" } }
```

Users who do not want a build step use `view()` directly. Both authoring paths compile to identical output — one implementation, two surfaces.

---

### Function components

Components are plain functions: `(props) => ViewNode`. No classes, no lifecycle methods, no special base types.

```ts
interface ButtonProps {
    label: () => string;
    disabled?: () => boolean;
    onClick: () => void;
}

function Button(props: ButtonProps): ViewNode {
    return view(
        "button",
        {
            disabled: props.disabled?.() ?? false,
            onClick: props.onClick,
        },
        props.label(),
    );
}
```

Props are reactive getter functions where reactivity is needed. Static props are plain values. The distinction is explicit at the call site — reactive values announce themselves.

---

### Lifecycle via `Scope` — no new API

Components run inside a `Scope`. `defer()` and `acquire()` work as-is for cleanup and resource management. No `onMount`/`onCleanup` aliases are introduced — wrapping existing primitives would obscure scope interoperability and sub-scoping patterns that users already know.

```ts
function Timer(): ViewNode {
    const count = Signal.create(0);
    const id = setInterval(() => count.set(count.get() + 1), 1000);
    defer(() => clearInterval(id));
    return view("div", {}, () => String(count.get()));
}
```

---

### Reactivity in templates — explicit calls

Reactive values are called explicitly inside `view()`. There is no auto-unwrapping, no special syntax. A reactive value is visually distinct from a static one:

```ts
// reactive — updates when name() changes
view("span", {}, name());

// static — set once at creation
view("span", {}, "hello");
```

Dynamic children (conditionals, expressions) are wrapped as getter functions. The JSX transform handles this automatically; direct `view()` callers wrap manually:

```ts
// conditional — re-evaluates when active() changes
view("div", {}, () => (active() ? view(Online) : view(Offline)));
```

---

### Control flow — no special components

No `<Show>`, no `<For>`. The existing primitives are sufficient:

- **Conditionals** — plain ternaries with explicit getter calls, wrapped in a function for the reactive boundary
- **Union branching** — `match()` is the natural primitive; exhaustive dispatch over variants is already the Aljabr pattern
- **Lists** — `ReactiveArray` children are detected by the renderer and reconciled surgically using the existing key-based diffing

```ts
// union-driven branching
view("div", {}, () =>
    match(status.state(), {
        Online: () => view(OnlineIndicator),
        Offline: () => view(OfflineIndicator),
        Reconnecting: ({ attempt }) =>
            view(Spinner, { label: () => `Attempt ${attempt}` }),
    }),
);

// list from ReactiveArray — surgical per-index updates
const rows = items.map((item) => view("li", {}, () => item.get("name")));
view("ul", {}, rows);
```

---

### Context — unchanged

The existing `context<T>()` from `aljabr/signals` works across component boundaries without modification. The owner tree is the component tree — `context.use()` walks it naturally.

**Implementation note:** the renderer must mount child component functions inside `runInContext(parentOwner, ...)` to keep the owner chain unbroken. This is a renderer implementation responsibility, not a public API concern.

---

### Module structure

```
aljabr/ui       ← createRenderer, view, ViewNode, RendererHost, RendererProtocol
aljabr/ui/dom   ← default DOM host implementation
```

The core `aljabr` package is untouched. Zero new runtime dependencies.

## Scope

| Feature                                                       | Status                                                       |
| ------------------------------------------------------------- | ------------------------------------------------------------ |
| `RendererHost<Node, Element>` interface                       | Included                                                     |
| `RendererProtocol` interface (batching escape hatch)          | Defined — not implemented in DOM host                        |
| `ViewNode` union (`Element`, `Text`, `Component`, `Fragment`) | Included                                                     |
| `createRenderer(host, protocol?)` factory                     | Included                                                     |
| `view()` factory + JSX transform support (`jsxImportSource`)  | Included                                                     |

| Reactive getter children (conditional branching)              | Included                                                     |
| `ReactiveArray` child reconciliation                          | Included                                                     |

| `@aljabr/react` adapter                                       | Deferred — contributor decision                              |

## Design Notes

### Components as unions — future consideration

An alternative component model was evaluated during design: components defined as tagged union variants, dispatched via `match()`. Under this model, the component tree is data — serializable, diffable, testable without a DOM.

This was deferred in favor of function components for ergonomic reasons. It is worth revisiting if the function component model shows limitations in practice, or if a data-first component model becomes desirable. Flagged for contributor discussion.
