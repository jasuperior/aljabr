# Aljabr Roadmap (v0.3.4): Native UI Layer — Full Implementation

_Minor release following v0.3.3. Completes the `aljabr/ui` layer introduced in v0.3.3: implements the `RendererProtocol` batching escape hatch in the DOM host, hardens edge cases, and polishes the authoring experience._

---

## Motivation

v0.3.3 ships the core architecture and a working DOM reference implementation. v0.3.4 completes it — the renderer protocol escape hatch, edge cases that surface during real use, and any authoring friction identified after v0.3.3 lands.

---

## Design Decisions

### `RendererProtocol` — DOM host implementation

The `RendererProtocol` interface was defined in v0.3.3 but not implemented in the DOM host. v0.3.4 wires it up — the DOM host accepts a custom `scheduleFlush` and uses it to batch reactive updates before flushing to the DOM.

Default behavior (no protocol provided) is unchanged: updates flush synchronously. Custom schedulers opt in explicitly:

```ts
// rAF batching — coalesces all updates within a frame into one flush
const { view, mount } = createRenderer(domHost, {
  scheduleFlush(flush) {
    requestAnimationFrame(flush)
  }
})
```

This closes the loop on the `signal` / `signal.protocol` analogy: simple path for most users, escape hatch for performance-sensitive targets.

---

### Edge cases and hardening

Cases deferred from v0.3.3 that surface during real use:

- Keyed fragment reconciliation — stable identity across conditional swaps
- Nested `ReactiveArray` children — arrays within arrays
- Prop diffing — avoid redundant `setProperty` calls when a reactive prop re-evaluates to the same value
- `null` / `undefined` children — treated as empty, not errors
- Component disposal ordering — LIFO guarantee across nested scopes

---

### Authoring polish

Any friction identified after v0.3.3 ships. Candidates:

- TypeScript inference for JSX props (narrowing component prop types through `view()`)
- Dev-mode warnings for common mistakes (missing reactive wrapper on dynamic children, unowned signal creation inside `view()`)
- Improved error messages for malformed `ViewNode` structures
