### **Aljabr Roadmap (v0.3.8): Canvas Renderer**

**Strategic Vision**

With the reactive primitive layer (`aljabr/prelude`, `aljabr/signals`) and the DOM renderer (`aljabr/ui/dom`) shipping in prior versions, v0.3.8 introduces a first-class canvas renderer at `aljabr/ui/canvas`. The goal is a fully pluggable, retained-mode canvas host that implements the existing `RendererHost<N, E>` contract — requiring zero changes to the core reconciler — and ships alongside a JSX namespace, a `Viewport` factory for pan/zoom, and an inherited paint property system via the context API.

The canvas renderer is a pure library export with no external dependencies. Authors building diagramming tools, data visualization plugins, or canvas-based UIs get a reactive, JSX-driven scene graph that integrates naturally with the rest of the Aljabr reactive stack. The diagramming plugin that motivated this work is an application-layer concern; the renderer itself stays general-purpose.

### Phase 2: Canvas Node Model & Host Skeleton

_Define `CanvasNode` as an aljabr union — parallel to `ViewNode` — and implement the `RendererHost<CanvasNode, CanvasElementNode>` interface._

#### 2.1 — `CanvasNode` as an Aljabr Union

`CanvasNode` follows the same pattern as `ViewNode`: named `Variant<>` types per variant, an internal `union()` factory, and a typed façade constant that shares the type name.

```ts
// src/ui/canvas/node.ts

import { union, type Variant } from "aljabr";

export type CanvasTag =
    | "rect"
    | "circle"
    | "ellipse"
    | "line"
    | "path"
    | "group"
    | "text";

export interface CanvasBounds {
    x: number;
    y: number;
    width: number;
    height: number;
}

// ---------------------------------------------------------------------------
// Variant payload types
// ---------------------------------------------------------------------------

export type CanvasElementNode = Variant<
    "Element",
    {
        tag: CanvasTag;
        props: Record<string, unknown>;
        children: CanvasNode[];
        parent: CanvasElementNode | null;
        bounds: CanvasBounds; // eager — recomputed on geometry prop change
        zIndex: number; // defaults to 0
        hitTest?: (x: number, y: number) => boolean; // pixel-perfect opt-in
    }
>;

export type CanvasTextNode = Variant<"Text", { content: string }>;

// ---------------------------------------------------------------------------
// CanvasNode — the union type
// ---------------------------------------------------------------------------

export type CanvasNode = CanvasElementNode | CanvasTextNode;

// ---------------------------------------------------------------------------
// Internal factory
// ---------------------------------------------------------------------------

const _canvasFactory = union({
    Element: (p: {
        tag: CanvasTag;
        props: Record<string, unknown>;
        children: any[];
        parent: CanvasElementNode | null;
        bounds: CanvasBounds;
        zIndex: number;
        hitTest?: (x: number, y: number) => boolean;
    }) => p,
    Text: (content: string) => ({ content }),
});

// ---------------------------------------------------------------------------
// CanvasNode — typed façade (value and type share the name)
// ---------------------------------------------------------------------------

export const CanvasNode = _canvasFactory as unknown as {
    Element(p: {
        tag: CanvasTag;
        props: Record<string, unknown>;
        children: CanvasNode[];
        parent: CanvasElementNode | null;
        bounds: CanvasBounds;
        zIndex: number;
        hitTest?: (x: number, y: number) => boolean;
    }): CanvasElementNode;
    Text(content: string): CanvasTextNode;
};
```

`CanvasBounds` is recomputed eagerly in `setProperty` whenever a geometry prop changes (`x`, `y`, `width`, `height`, `r`, `rx`, `ry`, `x1`, `y1`, `x2`, `y2`). For `path` elements, bounds are derived from the path bounding box via a lightweight parser. For `group` elements, bounds are the union of all descendant bounds — computed lazily and invalidated when any descendant geometry prop changes.

Because `CanvasNode` is an aljabr union, the reconciler and paint pass can use `match` to dispatch on variant — the same exhaustive pattern matching used throughout the rest of the library.

#### 2.2 — `RendererHost<CanvasNode, CanvasElementNode>` Implementation

The host is typed as `RendererHost<CanvasNode, CanvasElementNode>` — `CanvasElementNode` (not the full union) is the `E` parameter, mirroring how the DOM host is `RendererHost<Node, Element>`.

Implement all required host methods:

| Method                           | Canvas behaviour                                                                                                        |
| -------------------------------- | ----------------------------------------------------------------------------------------------------------------------- |
| `createElement(tag)`             | Returns `CanvasNode.Element({ tag, props: {}, children: [], parent: null, bounds: zeroBounds(), zIndex: 0 })`           |
| `createText(text)`               | Returns `CanvasNode.Text(text)`                                                                                         |
| `insert(parent, child, anchor?)` | Appends `child` to `parent.children` before `anchor` if provided, otherwise at end                                      |
| `remove(child)`                  | Removes `child` from its parent's `children`; sets `child.parent` to `null` for `Element` nodes                         |
| `setProperty(el, key, value)`    | Assigns to `el.props[key]`; recomputes `el.bounds` if a geometry key changed; updates `el.zIndex` if `key === "zIndex"` |
| `setText(node, text)`            | Matched via `match(node, { Text: (n) => n.content = text, Element: () => {} })`                                         |
| `parentNode(node)`               | `match(node, { Element: (n) => n.parent, Text: () => null })`                                                           |
| `nextSibling(node)`              | Walks `parent.children` to find the node after this one                                                                 |

`onMount`, `onUnmount`, `onUpdate` are not implemented in Phase 2 — canvas elements have no meaningful lifecycle hooks at the host level.

### Phase 3: Paint Pass & Primitive Rendering

_Implement the full repaint loop: clear the canvas, walk the scene graph, paint each primitive._

#### 3.1 — Paint Loop

The paint function is called by the `RendererProtocol` flush (Phase 4). It receives the `CanvasElementNode` root and a `CanvasRenderingContext2D`. Dispatch is via `match` on the `CanvasNode` union:

```
paintNode(ctx, node):
    match(node, {
        Text: () => return,              (bare text handled by parent's implicit wrap)
        Element: (el) =>
            ctx.save()
            applyTransform(ctx, el.props)    (x, y, scale, rotate for group)
            sortChildrenByZIndex(el.children)
            paintShape(ctx, el)
            for child of el.children:
                paintNode(ctx, child)
            ctx.restore()
    })
```

Children are sorted by `zIndex` in place during the paint pass. Insertion order is the tiebreaker for equal `zIndex` values.

#### 3.2 — Primitive Paint Implementations

Each `CanvasTag` maps to a set of `ctx` calls:

| Tag       | Canvas calls                                         | Key props                                                                                        |
| --------- | ---------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| `rect`    | `ctx.fillRect` / `ctx.strokeRect` or `ctx.roundRect` | `x`, `y`, `width`, `height`, `rx`, `fill`, `stroke`, `strokeWidth`                               |
| `circle`  | `ctx.arc` (full circle)                              | `cx`, `cy`, `r`, `fill`, `stroke`, `strokeWidth`                                                 |
| `ellipse` | `ctx.ellipse`                                        | `cx`, `cy`, `rx`, `ry`, `fill`, `stroke`, `strokeWidth`                                          |
| `line`    | `ctx.moveTo` + `ctx.lineTo`                          | `x1`, `y1`, `x2`, `y2`, `stroke`, `strokeWidth`, `lineCap`                                       |
| `path`    | `ctx.Path2D`                                         | `d` (SVG path string), `fill`, `stroke`, `strokeWidth`                                           |
| `group`   | transform only — no shape paint                      | `x`, `y`, `scale`, `rotate`                                                                      |
| `text`    | `ctx.fillText`                                       | `x`, `y`, `content`, `fontFamily`, `fontSize`, `fontWeight`, `fill`, `textAlign`, `textBaseline` |

Fill and stroke semantics follow SVG conventions: `fill="none"` skips fill; `stroke="none"` skips stroke. Both default to `none` unless set.

#### 3.3 — Group Transforms

`applyTransform` applies structured transform props to the canvas context before painting children:

```ts
ctx.translate(x ?? 0, y ?? 0);
ctx.rotate(((rotate ?? 0) * Math.PI) / 180);
ctx.scale(scale ?? 1, scale ?? 1);
```

All transforms are relative — they compose with the parent's accumulated transform via `ctx.save()` / `ctx.restore()`. There is no matrix prop; authors compose transforms by nesting `<group>` elements.

### Phase 4: Viewport, Culling & rAF Batching

_Wire up the `RendererProtocol` to `requestAnimationFrame`, implement the `Viewport` factory, and skip painting off-screen nodes._

#### 4.1 — rAF Protocol

```ts
const rafProtocol: RendererProtocol = {
    scheduleFlush(flush) {
        requestAnimationFrame(flush);
    },
};
```

#### 4.2 — `createCanvasRenderer`

A convenience wrapper that pre-wires `canvasHost` with `rafProtocol`:

```ts
export function createCanvasRenderer(canvas: HTMLCanvasElement) {
    const ctx = canvas.getContext("2d")!;
    const { view, mount } = createRenderer(canvasHost, rafProtocol);
    return {
        view,
        mount: (component: () => ViewNode) => {
            const unmount = mount(component, createRootElement(canvas, ctx));
            return unmount;
        },
    };
}
```

`createRootElement` allocates a synthetic root `CanvasElementNode` (via `CanvasNode.Element(...)`) that holds a reference to `ctx` and the canvas dimensions. The paint loop reads `ctx` and dimensions from this root.

#### 4.3 — `Viewport` Factory

`Viewport` is a factory (not a hook — consistent with the `Scope` naming convention) that owns pan/zoom state as signals and exposes the viewport bounds needed for culling:

```ts
interface ViewportHandle {
    x: Signal<number>;
    y: Signal<number>;
    scale: Signal<number>;
    bounds(): CanvasBounds; // derived — current visible world-space rect
    reset(): void;
}

export function Viewport(canvas: HTMLCanvasElement): ViewportHandle;
```

The `bounds()` derived is what the paint loop uses for culling. Authors feed `x`, `y`, `scale` into a root `<group>` as reactive props:

```tsx
const vp = Viewport(canvasEl);

<group x={vp.x} y={vp.y} scale={vp.scale}>
    {/* all diagram content */}
</group>;
```

#### 4.4 — Viewport Culling

Before painting each `CanvasElementNode`, the paint loop intersects `node.bounds` with the current viewport world-space bounds. If there is no intersection the node and its entire subtree are skipped:

```
intersects(node.bounds, viewport.bounds()) → false: skip
```

Bounds are in world space (pre-transform). For `group` elements the bounds are the union of all descendant bounds — computed lazily, cached, invalidated when any descendant geometry prop changes.

### Phase 5: Inherited Paint Props & Text Layout

_Implement context-based inheritance for font and color props, text layout positioning, and implicit string-child wrapping._

#### 5.1 — Inherited Paint Props via Context

Font and color props propagate down the scene graph using the existing `context()` API from `aljabr/signals`. A `PaintContext` holds the current inherited values and is provided at each `<group>` that sets any of these props:

Inheritable props:

| Prop          | Default        |
| ------------- | -------------- |
| `fontFamily`  | `"sans-serif"` |
| `fontSize`    | `14`           |
| `fontWeight`  | `"normal"`     |
| `fill`        | `"none"`       |
| `stroke`      | `"none"`       |
| `strokeWidth` | `1`            |

During the paint pass, each node reads the resolved value for a prop by checking `node.props[key]` first, falling back to the inherited context value. The context is updated at `group` boundaries when the group's props override any inheritable key.

#### 5.2 — Text Layout Props

The following props control how an implicit or explicit `<text>` child is positioned within a parent element's bounds. They are also inheritable via context:

| Prop            | Values                                               | Default  |
| --------------- | ---------------------------------------------------- | -------- |
| `textAlign`     | `"left"` \| `"center"` \| `"right"`                  | `"left"` |
| `verticalAlign` | `"top"` \| `"middle"` \| `"bottom"`                  | `"top"`  |
| `padding`       | `number` (uniform) or `{ top, right, bottom, left }` | `0`      |

When a `<text>` element is rendered as a child of a shape element, its `x` and `y` are computed from the parent's bounds + layout props rather than taken from its own props:

```
contentX:
    "left"   → parent.bounds.x + padding.left
    "center" → parent.bounds.x + parent.bounds.width / 2
    "right"  → parent.bounds.x + parent.bounds.width - padding.right

contentY:
    "top"    → parent.bounds.y + padding.top + fontSize (baseline offset)
    "middle" → parent.bounds.y + parent.bounds.height / 2
    "bottom" → parent.bounds.y + parent.bounds.height - padding.bottom

ctx.textAlign    = textAlign
ctx.textBaseline = "middle" for verticalAlign "middle", else "alphabetic"
```

#### 5.3 — Implicit Text Wrapping

When the reconciler encounters a `CanvasTextNode` (a `CanvasNode.Text(...)` variant) as a direct child of a non-`text` `CanvasElementNode`, the canvas host's `insert()` method automatically wraps it in a synthetic `CanvasNode.Element({ tag: "text", ... })` inheriting the parent's layout context. The synthetic element is internal — it does not appear in the scene graph as an authored node, only as a render-time detail.

In dev mode, the host warns if a `CanvasTextNode` is inserted into a `path` or `line` element (which have no meaningful bounding box for layout).

Word wrapping is explicitly out of scope. Authors requiring multi-line text should implement wrapping as a component that emits multiple `<text>` elements, or integrate an external library (e.g. Pretext).

### Phase 6: Hit Testing & Event Dispatch

_Implement pointer event handling: bounding-box hit tests, scene-graph bubbling, and the `on*` prop dispatch convention._

#### 6.1 — Event Listener Setup

`createCanvasRenderer` attaches a single set of pointer event listeners to the `<canvas>` DOM element during mount:

```
pointerdown, pointerup, pointermove, pointerenter, pointerleave, click, dblclick, contextmenu, wheel
```

Each listener extracts `(offsetX, offsetY)` from the native event (already in canvas coordinate space) and initiates a hit test against the scene graph.

#### 6.2 — Bounding-Box Hit Test

The hit test walks the scene graph in **reverse paint order** (last painted = topmost = checked first). `CanvasTextNode` variants are skipped — only `CanvasElementNode` variants participate in hit testing. For each `CanvasElementNode`:

1. Check `node.bounds` contains `(x, y)` in world space (apply accumulated inverse transforms)
2. If the node has a `hitTest` function, call it for pixel-perfect verification
3. First node that passes is the hit target

Groups are transparent to hit testing — a pointer event passes through a group to its children, and bubbles back up through the group after the deepest hit is found.

#### 6.3 — Event Bubbling

Starting from the deepest hit node, dispatch the event upward through `parent` pointers until the root is reached or a handler calls `event.stopPropagation()`. Each node's `on*` prop matching the event type is called:

```
onPointerDown → "pointerdown"
onClick       → "click"
onWheel       → "wheel"
... etc.
```

The synthetic canvas event passed to handlers mirrors the native `PointerEvent` interface with an additional `target: CanvasElementNode` field. `stopPropagation()` halts the bubble; `preventDefault()` is forwarded to the native event.

#### 6.4 — Pixel-Perfect Opt-In

Authors can attach a `hitTest` function directly on any element for irregular shapes:

```tsx
<path d={arcPath} hitTest={(x, y) => ctx.isPointInPath(path2D, x, y)} />
```

The offscreen canvas approach (unique-color rendering per node) is an alternative implementation strategy for the `hitTest` override — authors may use it, but the renderer does not provide it as a built-in. The `hitTest` contract is simply `(x: number, y: number) => boolean`.

### Phase 7: Export Surface & Canvas JSX Runtime

_Finalize the public API: the canvas JSX namespace, the export surface of `aljabr/ui/canvas`, and end-to-end integration testing._

#### 7.1 — Canvas JSX Runtime

`src/ui/canvas/jsx-runtime.ts` exports `jsx`, `jsxs`, and the `JSX` namespace. The factory is identical to the DOM version (`_jsx` → `view()`). The namespace declares canvas-specific intrinsic elements:

```ts
declare namespace JSX {
    interface IntrinsicElements {
        rect: RectProps;
        circle: CircleProps;
        ellipse: EllipseProps;
        line: LineProps;
        path: PathProps;
        group: GroupProps;
        text: TextProps;
    }
}
```

Each props interface is typed with both geometry props and the full set of inheritable paint props (`fill`, `stroke`, `fontFamily`, etc.) as optional keys. `on*` event handlers accept `(event: CanvasSyntheticEvent) => void`.

Authors set `jsxImportSource: "aljabr/ui/canvas"` in their `tsconfig.json` (or use a per-file `/** @jsxImportSource aljabr/ui/canvas */` pragma) for canvas component files. DOM and canvas component files use separate `jsxImportSource` values — the boundary type at interop points is `ViewNode`, which both runtimes produce.

#### 7.2 — Export Surface of `aljabr/ui/canvas`

`src/ui/canvas/index.ts` exports:

```ts
export { canvasHost }; // RendererHost<CanvasNode, CanvasElementNode>
export { createCanvasRenderer }; // pre-wired with rAF protocol
export { Viewport }; // pan/zoom factory
export { CanvasNode }; // union façade — CanvasNode.Element(...), CanvasNode.Text(...)
export type {
    CanvasNode,
    CanvasElementNode,
    CanvasTextNode,
    CanvasBounds,
    CanvasTag,
};
export type { ViewportHandle };
```

Not exported: internal paint utilities, the rAF protocol object (authors who need a custom protocol use `createRenderer(canvasHost, myProtocol)` directly from `aljabr/ui`).

#### 7.3 — Implementation Order

1. Phase 1: module scaffolding + DOM JSX rename
2. Phase 2: type definitions + host skeleton (all methods stubbed with no-ops)
3. Phase 3: paint loop + all primitive renderers
4. Phase 4: rAF protocol + `Viewport` + viewport culling
5. Phase 5: paint prop inheritance + text layout + implicit wrapping
6. Phase 6: hit testing + event bubbling + `on*` dispatch
7. Phase 7: JSX runtime + export surface + integration tests

Tests live in `test/ui/canvas/`, one file per phase concern. Each phase ships with tests before the next begins.

### Design Constraints

- **Zero external dependencies.** The canvas renderer uses only the browser Canvas API and existing `aljabr` internals.
- **No layout engine.** Single-line text layout only. Word wrap is an application concern.
- **No word wrap.** Authors integrate external libraries (e.g. Pretext) as needed.
- **Renderer-agnostic `ViewNode`.** The canvas renderer produces and consumes the same `ViewNode` / `Child` types as the DOM renderer. No new core types.
- **`createPortal` is not part of this release.** Portal-style cross-renderer composition is a userland pattern (spawn a new renderer root, tie its cleanup to the parent owner). If the pattern proves sound in application code, it will be promoted to `aljabr/ui` in a future version.
