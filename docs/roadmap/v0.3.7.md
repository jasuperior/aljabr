# Aljabr Roadmap (v0.3.7): Iterator Reactivity & List Reconciliation

_Patch release following v0.3.6. Completes the reactive list reconciliation work deferred since v0.3.4: per-index scopes in the renderer, keyed DOM reconciliation, and key propagation through all six iterator methods that produce a `DerivedArray`. Also fixes the local dev environment._

---

## Motivation

Since v0.3.4, `mountDerivedArray` in the renderer has carried a documented deferral:

```
// Fine-grained per-index updates are left for v0.3.4.
```

The consequence is visible in real application code: any mutation to a reactive list — toggle, add, remove, filter change — tears down and re-mounts the **entire** rendered list. Every item re-renders on every change, regardless of whether its data changed.

This release closes that gap completely. The fix spans two layers: the `DerivedArray` data model (key propagation through the iterator chain) and the renderer (per-index scopes, then keyed reconciliation). All six methods that produce a `DerivedArray` are addressed — none are left behind.

---

## Design Decisions

### Key propagation through the iterator chain

`DerivedArray` and `RefArray` each expose three iterator methods that return a new `DerivedArray`: `map`, `filter`, and `sort`. Key information must survive the entire chain for the renderer to reconcile lists surgically. The current state of each method:

| Method | Source | Current key handling | Gap |
|---|---|---|---|
| `RefArray.map()` | mutable root | `keyFn = null`, no `opts` | No way to inject a key |
| `RefArray.filter()` | mutable root | `opts?.key ?? identity` | None — root source, nothing to inherit |
| `RefArray.sort()` | mutable root | `opts?.key ?? identity` | None — root source, nothing to inherit |
| `DerivedArray.map()` | derived | `keyFn = null` | Source key silently dropped |
| `DerivedArray.filter()` | derived | `opts?.key ?? identity` | Falls back to identity when source has a key |
| `DerivedArray.sort()` | derived | `opts?.key ?? identity` | Falls back to identity when source has a key |

**`DerivedArray.filter()` and `DerivedArray.sort()` — inherit source key**

When no `opts.key` is provided and the source `DerivedArray` already has a `#keyFn`, inherit it instead of defaulting to reference identity. The items produced by filter and sort are a subset or reorder of the source items, so the source key function is valid for them.

```ts
// before — key lost on chained filter
const done = visibleTasks.filter(t => isDone(t));        // keyFn = identity (wrong)

// after — key inherited from visibleTasks
const done = visibleTasks.filter(t => isDone(t));        // keyFn = taskId (inherited)
```

`keyIsDefault` is cleared when a key is inherited this way — no warning is emitted.

**`DerivedArray.map()` — index-based key inheritance**

`map` changes the element type (`T → U`), so the source key function cannot operate on the mapped items directly. Instead, `map` inherits the source key as an index-based lookup: a closure over the source array that resolves `sourceKeyFn(source.peek(i))` for index `i`. The resulting `DerivedArray<U>` carries this index key and exposes it via `keyAt(i)`.

```ts
// key is taskId on visibleTasks; rows inherits it index-by-index
const rows = visibleTasks.map(task => <TaskItem task={task} />);
rows.keyAt(0); // → taskId(visibleTasks.peek(0))
```

**`RefArray.map()` — add `opts` parameter**

`RefArray` is the mutable root; it has no key to inherit. Authors mapping directly off a `RefArray` (without a prior `filter` or `sort` step) have no current way to inject a key. Add `opts?: { key: (item: T) => unknown }` to `RefArray.map()` to close this gap.

```ts
const rows = tasks.map(task => <TaskItem task={task} />, { key: taskId });
```

**`keyAt(i): unknown | null` — public key surface on `DerivedArray`**

Add a single public method that the renderer uses to retrieve the key for any index. Implementation:

- For `filter`/`sort` arrays: `#keyFn(#items[i])`.
- For `map` arrays with an inherited source key: delegates to the stored index-based closure.
- For `map` arrays with no key (e.g. mapping off an unkeyed `RefArray`): returns `null`.

`null` signals the renderer to fall back to position-based reconciliation for that array.

### Renderer — Phase A: per-index scopes (position-based)

Replace the current `mountDerivedArray` implementation, which subscribes to all signals in a single `effectOwner` and tears down the entire list on any change, with per-index owner scopes.

Each index `i` gets its own `Computation` that subscribes only to `arr.get(i)`. The outer `effectOwner` subscribes only to `arr.length()`:

- **Length increases**: create new per-index scopes for the appended indices.
- **Length decreases**: dispose trailing scopes; their cleanups remove the corresponding DOM nodes.
- **`arr.get(i)` changes**: only the scope at index `i` re-runs, reconciling the old `Child` at that position with the new one.

This eliminates full-list re-renders for the common cases: in-place mutation (toggle), append (add), and tail removal (remove last).

---

### Renderer — Phase B: keyed scopes

Per-index scopes (Phase A) handle structural changes by position. When items are reordered or removed from the middle of a filtered list, items shift positions: their per-index signals fire and they re-render even though their data is unchanged.

Keyed scopes bind the scope's lifetime to item identity rather than position.

The renderer checks `arr.keyAt(i)` on the first render and after each length change to determine whether to use keyed or position-based mode:

- If `keyAt` returns non-null values: **keyed mode**.
- If `keyAt` returns null: **position-based mode** (Phase A).

**Keyed mode implementation:**

Maintain `Map<unknown, { scope: Computation; start: N; end: N }>` — one entry per live item, keyed by the value returned from `keyAt`.

On each update (triggered by the length signal or root signal):

1. Compute old key list and new key list.
2. **Removed keys**: dispose the scope; remove all DOM nodes between its `start` and `end` anchors.
3. **New keys**: create a fresh scope; insert its `start`/`end` anchors and reconcile the item at the correct position.
4. **Moved keys** (same key, new position): re-insert the existing `start`/`end` anchors and all nodes between them at the new position in the DOM — without disposing or re-running the scope.
5. **Same key, changed value** (`keyAt(i)` unchanged but `arr.get(i)` reference changed): the per-index signal fires and the scope re-runs at its current position.

The result: a task toggled from Active to Done re-renders only its own row. A filter change that removes one item from the middle of the list moves the remaining items' DOM nodes without re-rendering them.

## Future Work

### Keyed portal / cross-renderer composition

Keyed reconciliation within a single renderer is complete after this release. Composition across renderers (e.g. a DOM list item that contains a canvas sub-renderer) is a userland pattern today — spawn a new renderer root, tie its cleanup to the parent owner. If the pattern proves common in application code, a first-class `createPortal` primitive will be considered for `aljabr/ui` in a future version.
