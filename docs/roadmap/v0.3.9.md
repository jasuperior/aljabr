### **Aljabr Roadmap (v0.3.9): Canvas Bounds Completeness**

**Strategic Vision**

v0.3.8 shipped a canvas renderer with three deliberate bounds deferrals тАФ `<path>`, `<group>`, and `<text>` all return `zeroBounds()` from the host's `recomputeBounds` switch. The paint pass and hit-test walker compensate via the `hasCullableBounds(b) тЗТ b.width > 0 && b.height > 0` rule and the `onHitTest`-is-authoritative escape hatch from ┬з6.4: zero-area bounds always paint (no culling) and only-the-override-decides-the-hit. That tradeoff is functionally correct but optimisation-visible тАФ large scenes pay per-child intersect tests instead of group-level skips, and pixel-perfect hit testing must be hand-rolled by the author for every path or label they want clickable.

v0.3.9 lifts those three deferrals so every primitive carries meaningful bounds. Once each tag grows real bounds, the existing `intersects()` / `boundsContains()` checks engage automatically тАФ there are no new code paths in `paint.ts` or `hit-test.ts`. This is a tightly-scoped completeness release; no new user-facing features.

The renderer's external surface (`aljabr/ui/canvas/index.ts`) does not change. Only `host.ts`'s `recomputeBounds` and the per-primitive bounds modules grow code.

### Phase 1: Path Bounds

#### 1.1 тАФ The Deferral

`src/ui/canvas/host.ts`'s `recomputeBounds` returns `zeroBounds()` for `<path>` elements:

```ts
when({ tag: "path" }, () => zeroBounds()),
```

Consequences in production code:

- **Culling**: `paint.ts`'s `hasCullableBounds` short-circuits paths to "always paint" тАФ viewport culling never trims a path subtree even when it's miles off-screen.
- **Hit testing**: `hit-test.ts`'s `hitTestNode` only finds a path when the author provides an `onHitTest` override (Phase 7's authoritative-override fix). A bare `<path d="тАж">` is unhittable by bounds alone.

The `<path>` integration test in `test/ui/canvas/integration.test.ts` exists exactly because of this gap тАФ it pins the override-path semantics so the gap can be lifted without regression.

#### 1.2 тАФ Motivations for Deferral

SVG path strings carry the full grammar: `M`, `L`, `H`, `V`, `C`, `S`, `Q`, `T`, `A`, `Z`, plus their relative-coordinate lowercase variants. A *correct* bounding box requires evaluating bezier extrema тАФ for a cubic curve, the bounds aren't `(min(p0, p1, p2, p3), max(...))` because the curve can swing outside its control points along the parameter `t`. The exact extrema solve `B'(t) = 0`, a small derivative-root problem per axis.

Phase 2 of v0.3.8 was structured around the union shape and the host skeleton тАФ landing a path-string parser at that point would have doubled the file's surface area for a feature the rest of v0.3.8 didn't strictly need (every other primitive computes its bounds from numeric props directly). The roadmap `┬з2.1` explicitly noted "For `path` elements, bounds are derived from the path bounding box via a lightweight parser" as a follow-on item, and the comment in `recomputeBounds` was kept honest: "Path (SVG path-string parsing) and group (union of descendants) bounds remain deferred."

#### 1.3 тАФ Implementation Strategy Options

| Option | Approach | Approx. LOC | Correctness | Notes |
|---|---|---:|---|---|
| 1 | **Endpoint-only union** тАФ parse only `M`/`L`/`H`/`V`/`Z` start/end points; treat curves as straight lines. | ~30 | Under-paints curves (curve arcs may extend outside the endpoint hull). | Cheapest; only correct for poly-line paths. **Rejected** тАФ under-painting silently hides geometry, which is the one failure mode we should never ship. |
| 2 | **Control-point hull** тАФ parse all command operands (including bezier / quadratic control points and arc box corners), take min/max of every reference point. | ~50 | Always correct; can overestimate by the curve's hull-vs-curve gap. | Good balance: never under-paints, modest code, no math beyond min/max. Same model as SVG's own bounding-box recommendations for editors that want a fast-path. |
| 3 | **Bezier extrema** тАФ solve `B'(t) = 0` per axis to find tight bounds for each curve segment. | ~150 | Tight bounds. | More math, more allocation per geometry change. Worth it if a future profile shows the hull's overestimate is the bottleneck. |
| 4 | **`Path2D` + `OffscreenCanvas` rasterisation** тАФ construct `new Path2D(d)`, render to a 1├Ч1 offscreen canvas, query the resulting transform extent. | ~40 | Tight bounds via the browser's own rasteriser. | Browser-only (no Node fallback for any future SSR experiments); brittle if Path2D bounding semantics shift; needs an offscreen canvas allocation per measurement. |

**Recommendation: Option 2 (control-point hull).** Best balance of correctness, code size, and zero external dependencies. The overestimate is bounded by the curve's hull, which for typical diagramming paths (rounded corners, gentle bezier connectors) is well within an order of magnitude of the tight bounds тАФ culling still works, hit testing falls through to `onHitTest` for shapes where the hull is wildly off (a thin S-curve).

If a profile-driven need for tighter bounds materialises, Option 3 slots in as a successor without changing the public API тАФ `recomputeBounds` is the only caller.

#### 1.4 тАФ Acceptance Criteria

- A `<path d="M0 0 L100 100">` reports `bounds = { x: 0, y: 0, width: 100, height: 100 }` after `setProperty("d", тАж)`.
- The Phase 7 integration test for `onHitTest` rejection still passes (the override is consulted; the new bounds path is the *fallback*).
- A new test covers a curved path (`M0 0 C50 -50 100 50 150 0`) and asserts the control-point-hull bounds.
- The "always-paints regardless of viewport" test in `test/ui/canvas/culling.test.ts` for paths flips to assert culling now engages, with a comment explaining the v0.3.9 promotion.

### Phase 2: Group Bounds (Descendant Union)

#### 2.1 тАФ The Deferral

`recomputeBounds` for `<group>` returns `zeroBounds()`, mirroring the path situation. The paint pass and hit-test walker recurse through groups regardless of viewport intersection (zero-area = always paint), then cull each child individually. v0.3.8 ┬з4.4 specifies the eventual behaviour:

> For `group` elements the bounds are the union of all descendant bounds тАФ computed lazily, cached, invalidated when any descendant geometry prop changes.

Currently nothing computes that union. The per-child intersect path is correct but pays `O(N_descendants)` rather than `O(1)` for any group whose subtree is entirely off-screen.

#### 2.2 тАФ Motivations for Deferral

The interesting work isn't the union itself тАФ it's *invalidation*. A correct lazy cache requires:

1. Every `setProperty` on a primitive that mutates a geometry key walks up its `parent` chain and marks each ancestor `<group>` dirty.
2. Every `insert` and `remove` invalidates the new and old parent `<group>`s up their respective chains.
3. The dirty bit clears on read of `el.bounds` (or whenever the group's bounds are recomputed).

This is a classic cache-invalidation problem and the test surface for it is wide: prop changes, list reorders (keyed `RefArray.map` reorders touch parent pointers), un-mounts, scope-driven cleanup. v0.3.8 already shipped two side maps for the implicit text wrapping (`textWrappers` / `wrapperToText`); adding a third state machine for group-bounds dirtiness тАФ without first having a paint-pass *bottleneck* that justifies it тАФ felt like premature optimisation. The fall-through to "recurse and child-cull" works correctly today.

#### 2.3 тАФ Implementation Strategy Options

| Option | Approach | Approx. LOC | Notes |
|---|---|---:|---|
| 1 | **Eager** тАФ recompute the union in `setProperty` and `insert`/`remove`, walking the parent chain bottom-up. | ~30 | Simplest. Cost is paid every prop write; expensive for groups with thousands of descendants and frequent geometry mutations. |
| 2 | **Lazy with a dirty bit** (per `┬з4.4`) тАФ `setProperty`/`insert`/`remove` mark the chain dirty; `el.bounds` is recomputed on the first read after a dirty mark and cached. | ~50 | Matches the spec. Amortised O(1) per write; cost paid once per frame on the read side (paint pass / hit test). |
| 3 | **Batched recomputation at flush boundary** тАФ collect dirty groups during the rAF tick; recompute them once before paint reads the bounds. | ~40 | Marries to the existing rAF protocol; one well-defined recompute pass per frame. Slightly more orchestration. |
| 4 | **Skip the optimisation entirely** тАФ leave child-by-child culling in place. | 0 | The status quo. `O(N)` intersect tests stays acceptable for the few-thousand-element scenes the diagramming target needs. |

**Recommendation: Option 2.** It matches the v0.3.8 spec and the "compute when read" pattern used elsewhere in aljabr (cf. `Derived` in `prelude/derived.ts`, where a dirty notification triggers eventual re-evaluation, not an immediate one). The dirty bit lives on the `CanvasElementNode` payload тАФ no new variant, no new map.

If profiling shows that walking up the parent chain on every `setProperty` is itself the cost (deep trees, dense mutation), Option 3 becomes attractive тАФ but only with that profile in hand.

#### 2.4 тАФ Acceptance Criteria

- A `<group>` with a single far-off-screen child reports bounds equal to that child's bounds after the next read.
- Mutating a descendant primitive's `x` invalidates all ancestor groups' caches.
- Inserting / removing a child invalidates the immediate parent group and its ancestors.
- A new test in `test/ui/canvas/culling.test.ts` mounts a group with N children, places the entire group off-screen, and asserts that the paint pass emits zero `fillRect` calls (whereas the same scene under v0.3.8 emitted N-1 culled-child intersect tests).
- The reactive-list reorder test from `renderer.test.ts` keeps passing тАФ the dirty walks must compose with `RefArray.map`'s keyed re-parenting.

### Phase 3: Text Bounds

#### 3.1 тАФ The Deferral

`<text>` elements currently report `zeroBounds()`. The text layout pass (Phase 5) reads the *parent shape's* bounds when computing the wrapped text's `x`/`y`, so authored layouts work тАФ but the text element itself contributes nothing to its own subtree's bounds. Hit testing on a label requires the click to land inside the parent shape's bounds; a label that overflows its parent (intentionally or by font fallback) is unhittable past the parent's edge.

#### 3.2 тАФ Motivations for Deferral

Text bounds require font metrics. The only general-purpose source is `CanvasRenderingContext2D.measureText(content)`, which needs:

- A `ctx` reference (which `host.ts`'s `setProperty` doesn't have тАФ it operates on the scene graph, not on a particular renderer).
- `ctx.font` set to the resolved font string before measuring (`fontWeight fontSize fontFamily`), which itself depends on `PaintContext` resolution. At `setProperty` time, the paint context isn't yet known тАФ it's resolved during the paint walk.

The roadmap ┬з5.2 spelled out *layout* (positioning text inside a parent's bounds) but did not require text *measurement* to ship in v0.3.8 тАФ wrapping authoring against fixed shape parents avoids the question entirely. So text bounds were silently deferred along with path and group bounds.

#### 3.3 тАФ Implementation Strategy Options

| Option | Approach | Approx. LOC | Notes |
|---|---|---:|---|
| 1 | **Measure at paint time** тАФ `paintShape` calls `ctx.measureText(content)` after composing `ctx.font`, writes the result into `el.bounds` for the next frame. | ~20 | Bounds are correct *one frame late* for the first paint and after every font/content change. Hit tests that fire before the first paint of a fresh element see `zeroBounds`. Simplest path; works in practice for interaction-driven UI (paint always happens before the user clicks). |
| 2 | **Shared offscreen canvas** тАФ `host.ts` allocates a single `OffscreenCanvas` (or DOM `HTMLCanvasElement` fallback) and measures eagerly in `setProperty` whenever `content`, `fontFamily`, `fontSize`, or `fontWeight` changes. | ~40 | Bounds are always correct, even before the first paint. Resolves font through the same context resolution as paint, but at write time тАФ needs to walk up to find the inherited values, which the paint pass also does. Best general behaviour; modest startup cost (offscreen canvas allocation, one per process). |
| 3 | **Heuristic estimate** тАФ `width тЙИ fontSize ├Ч 0.6 ├Ч content.length`, `height тЙИ fontSize ├Ч 1.2`. | ~10 | No canvas needed, no measurement cost. Useful only for culling (overestimate is harmless); too imprecise for hit testing on proportional fonts. |
| 4 | **Defer to author** тАФ expose `measureText(text, ctx)` as a utility, let the diagramming application opt into setting a `bounds` prop manually. | ~5 | Punts the problem. Encourages the same workaround we already have for paths via `onHitTest`. |

**Recommendation: Option 2 (shared offscreen canvas).** It matches the "eager on `setProperty`" pattern already used by every other primitive's bounds тАФ there's no two-pass model elsewhere in the host, and introducing one for text alone (Option 1) creates a third state-machine class authors have to reason about. The offscreen canvas allocation is one-per-process; its cost is paid once.

The font-resolution piece тАФ walking up to find the nearest `<group>` that overrides `fontFamily`/`fontSize`/`fontWeight` тАФ already exists in `paint.ts`'s `deriveContext` walk; lifting the lookup into a shared helper is part of the work.

#### 3.4 тАФ Acceptance Criteria

- A `<text content="Hello" fontSize=20>` with no explicit `x`/`y` reports `bounds.width > 0` and `bounds.height тЙИ 20` after `setProperty("content", тАж)`.
- Changing `fontSize` re-measures and updates bounds.
- Changing `fontFamily` on an ancestor `<group>` triggers a re-measure on the descendant text (this requires the font-resolution helper to be re-runnable when ancestor context changes).
- A label that overflows its parent shape's right edge is hittable up to its actual rendered right edge.
- Existing text-layout tests continue to pass тАФ the *positioning* logic doesn't change, only the *bounds* computation gains substance.

### 4.0 Out of Scope (Tracked)

These surfaced during v0.3.8 but are intentionally not part of v0.3.9. Each is recorded so a future version can lift it without re-discovering the context.

#### 4.1 тАФ `CanvasNode` value-and-type re-export through the barrel

Today `aljabr/ui/canvas` exports the `CanvasNode` *type* but the *value* (the variant factory) lives only in `aljabr/ui/canvas/node`. This is forced by `verbatimModuleSyntax: true` rejecting a same-name `export { X }` + `export type { X }` pair through one barrel. v0.3.8 documented the workaround in `index.ts`. A future version could split `node.ts` into `node-types.ts` (types only) + `node-factory.ts` (value), letting both names re-export through the barrel without conflict тАФ at the cost of an extra file. Defer until someone actually needs the factory through the public surface (the host's `createElement` / `createText` cover all known authoring sites).

#### 4.2 тАФ `onHitTest` naming vs. the spec's original `hitTest`

v0.3.8 ┬з6.4 was revised to use `onHitTest` because the reconciler's `on*` rule routes callback functions through without the reactive-getter trap. The spec's original `hitTest` name would require either (a) a special-case in the core renderer's reactive-prop check, or (b) a function-arity heuristic. Defer until either becomes warranted; the current naming is consistent with the rest of the event-handler family.

#### 4.3 тАФ Word wrap for `<text>`

Explicitly out of scope per v0.3.8 ┬з5.3. Authors integrate Pretext or similar; the renderer continues to single-line. Lifting this would require a layout engine, which v0.3.x does not have an appetite for.

#### 4.4 тАФ `createPortal` for cross-renderer composition

Explicitly out of scope per v0.3.8 design constraints. Promotion to `aljabr/ui` waits on the diagramming-plugin author validating the userland pattern (mount a fresh renderer root, tie cleanup to the parent owner) in production. Once validated, a thin wrapper lands in core.

#### 4.5 тАФ Per-frame matrix allocation in `hit-test.ts`

`hit-test.ts`'s `compose` allocates a fresh tuple per group on every hit walk. For scenes with deep group nesting and heavy pointer traffic this could show up in profiles. A pooled-matrix approach would amortise the cost; defer until a profile demands it.

### Design Constraints

- **Public API unchanged.** No new exports from `aljabr/ui/canvas/index.ts`. No new prop names. No new `JSX.IntrinsicElements` keys.
- **No new variants on `CanvasNode`.** Bounds-completeness fits inside the existing payload via `bounds` writes.
- **No regressions in v0.3.8 tests.** Every existing test file under `test/ui/canvas/` continues to pass without edits, except for the explicit "always paints regardless of viewport" assertions for path/group/text тАФ those tests are flipped to assert the new culling behaviour, with a comment pointing at this roadmap.
- **No external dependencies introduced.** The path-string parser, group union walker, and offscreen-canvas measurement all use stdlib + browser APIs already in scope.

---

### Implementation Order

1. Phase 1: Path bounds (control-point hull parser).
2. Phase 2: Group bounds (lazy + dirty bit).
3. Phase 3: Text bounds (shared offscreen canvas).

Each phase ships with the test updates flagged in its acceptance criteria before the next begins. Phases 1 and 2 are independent and could ship in either order; Phase 3 depends only on the existing paint context resolution and is independent of the other two.
